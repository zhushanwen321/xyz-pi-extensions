# 5 业务追踪视角（Requirement Clarity）

> 独立 subagent 用。fresh context 下，按这 5 视角对 requirements.md 初稿做强制枚举追踪。
> 卡住的地方 = gap。
>
> **关键区别：** 这些是**业务视角**，不是实现视角。本阶段明确不考虑系统实现、技术栈选型。
> 如果追踪中发现实现层问题（API 契约、状态机、技术架构），记录为「移交②系统设计处理」，
> 不在本阶段展开。

## 追踪规则

1. **逐视角追踪**，每个视角必须核对或写降级理由（不适用时）
2. **卡住的地方 = gap**，标注类型（F/K/D）和具体问题
3. **YAGNI**：只追踪当前需求涉及的路径，不探索无关场景
4. **业务优先**：所有问题回到「业务目标是什么」「用户要什么」，不滑入技术实现

---

## 视角 1: Goal Traceability（目标可追溯性）

追踪从业务目标到达成路线到用例的完整可追溯链。

### 检查项

- [ ] 每个业务目标是否有对应的达成路线（策略/方法）？
- [ ] 每条达成路线是否拆解到了可验证的业务用例？
- [ ] 是否有「孤立用例」——存在但没有对应目标的用例？（可能目标遗漏或用例多余）
- [ ] 是否有「孤立目标」——声明了目标但没有路线也没有用例支撑？
- [ ] 目标之间是否有依赖或冲突关系？文档是否说明了？
- [ ] 成功标准是否可衡量？（不只是「做好 X」，而是「X 达到 Y 指标」）

### 典型 gap

- **K**：目标「提升用户留存」——留存指次日/7日/30日？当前基线是多少？
- **D**：目标「支持多租户」与「保持单租户简单」冲突——哪个优先？
- **F**：文档声称「现有系统已支持 X」，但追踪代码发现 X 只是个空壳

---

## 视角 2: Actor & Use Case Completeness（角色与用例完整性）

追踪所有参与角色及其用例是否完整。

### 检查项

- [ ] 是否列出了所有与系统交互的 Actor（不只是「用户」——还有管理员、外部系统、定时任务等）？
- [ ] 每个 Actor 的用例是否完整？是否有 Actor 列了但没行为？
- [ ] 用例图是否标注了系统边界（include/extend 关系）？
- [ ] 是否有「隐含 Actor」——被提到但没正式纳入的角色（如「审核人」「系统管理员」）？
- [ ] 同一 Actor 的不同权限级别是否区分？（普通用户 vs VIP，员工 vs 管理员）
- [ ] 用例之间是否有前置/后置关系？是否标注了？

### 强制检查项

对每个用例，必须回答：

- [ ] 主流程（happy path）是什么？
- [ ] 替代流程有哪些？
- [ ] 异常流程（Actor 中途退出、输入错误、权限不足）怎么处理？
- [ ] 用例的前置条件是什么？后置状态是什么？

### 典型 gap

- **K**：「用户」是指终端用户还是操作员？两者权限不同——都列出用例了吗？
- **D**：「批量操作」是独立用例还是「单条操作」的扩展？
- **F**：用例图标注了 3 个 Actor，但业务描述里还提到「代理商」这个角色

---

## 视角 3: Data Flow Completeness（数据流完整性）

追踪业务数据从产生到消费到归档的完整生命周期。

### 检查项

- [ ] 是否画了数据流图（DFD），标注了数据源、处理、存储、流向？
- [ ] 每类业务数据是否追踪了：谁产生→谁处理→谁消费→如何归档/销毁？
- [ ] 数据在系统间的流转是否标注了？（从 A 系统到 B 系统，格式是什么）
- [ ] 是否有「数据孤岛」——产生但没人消费的数据？
- [ ] 是否有「无源数据」——被消费但没人负责产生的数据？
- [ ] 数据的敏感级别是否标注？（公开/内部/机密——影响④非功能性设计）

### 典型 gap

- **K**：订单数据流向「财务系统」——财务系统是现有系统还是新建？数据格式约定是什么？
- **D**：用户行为日志——实时流处理还是批量 ETL？
- **F**：数据流图画了「报表数据从分析库来」，但分析库当前不存在

---

## 视角 4: UI/UX Scenario Coverage（界面交互场景覆盖）

追踪用户交互场景的完整性（线框/流程级别，不涉及技术实现）。

### 检查项

- [ ] 每个用户用例是否有对应的 UI 交互场景描述？
- [ ] 关键页面的线框/布局是否描述了？（可以是文字描述，不要求设计稿）
- [ ] 交互流程是否画了？（用户从哪进来→经过哪些页面→在哪完成→去哪）
- [ ] 是否考虑了不同终端？（桌面/移动/平板——如果适用）
- [ ] 空状态、加载状态、错误状态的交互是否描述了？
- [ ] 是否有「无 UI 的用例」——比如 API-only 的交互、定时任务触发？这些也要描述触发和结果

### 降级条件

纯后端/API 系统、纯技术重构、无用户交互界面的需求——本视角写降级理由「本需求无 UI 交互」，跳过。

### 典型 gap

- **K**：列表页默认排序是什么？分页还是无限滚动？
- **D**：表单提交成功后——跳转到详情页、显示 toast、还是留在原页？
- **K**：移动端的交互和桌面端是否一致？还是需要差异化设计？

---

## 视角 5: Cross-System Dependency（跨系统功能关联）

追踪系统间功能依赖和关联关系。

### 检查项

- [ ] 是否画了系统间功能关联图（哪些系统提供什么功能给本系统）？
- [ ] 本系统依赖的外部系统功能是否逐一列出？
- [ ] 本系统提供给其他系统的功能是否逐一列出？
- [ ] 系统间的依赖方向是否清晰？（谁依赖谁，避免循环依赖）
- [ ] 跨系统交互是同步还是异步？（影响后续的非功能性设计）
- [ ] 外部系统的契约是否稳定？（是自有系统可控，还是第三方不可控）

### 降级条件

单系统无外部依赖——本视角写降级理由「本需求无跨系统依赖」，跳过。

### 典型 gap

- **K**：依赖「支付系统」——是调用支付系统的 API 还是消息队列？超时怎么处理？
- **D**：本系统是否需要对外提供 webhook 通知？还是由对方轮询？
- **F**：关联图画了「消息队列 Kafka」，但追踪发现当前用的是 RabbitMQ

---

## gap 分流

追踪完成后，所有 gap 按 F/K/D 分类：

| 类型 | 含义 | 处理 |
|------|------|------|
| **F（Fact）** | 客观事实（代码/文档里有但初稿没提到的信息） | 主 agent 二次确认（可能误报） |
| **K（Knowledge）** | 主 agent 不知道的业务规则 | 直接问用户 |
| **D（Decision）** | 需要做权衡的业务决策 | 给方案对比 + trade-off |

详见 shared-loop.md 的 Step 3。

## 收敛判定

收敛复核轮次（除追踪外）还要执行：

- 如果追踪无新 gap → 在 tracing-round-{N}.md 顶部标注 `CONVERGED` 并列出已追踪的视角
- 如果有新 gap → 正常返回 gap 列表（不标 CONVERGED）
