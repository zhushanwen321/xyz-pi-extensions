Workflow 数据流分析

为什么 description 没有显示？信息是怎么获取的？

📦 数据源 A：脚本原始信息

来自 workflow 脚本文件（.pi/workflows/*.js）

config-loader.ts — extractMetaViaRegex()

meta.name string "review-fix-loop"
meta.description string "Review-fix loop: code-review → fix..."
meta.phases array [{title:"Review"}, {title:"Judge"}, ...]
存储位置
CachedWorkflowMeta — 内存缓存，按 workspace 分桶
通过 getWorkflow(name) 或 loadWorkflows() 读取

✅ 信息完整，始终可用

⚡ 数据源 B：执行运行时信息

来自 orchestrator 运行 workflow 后产生的实例

state.ts — WorkflowInstance

instance.name string "review-fix-loop"
instance.description string? undefined ← 旧实例没有！
instance.status WorkflowStatus "completed"
instance.trace ExecutionTraceNode[] [{agent, task, model, status, phase, ...}]
instance.worker string "/path/to/review-fix-loop.js"
存储位置
orchestrator.instances Map — 按 runId 索引
通过 orchestrator.getInstance(runId) 读取

⚠️ description 只在 run() 时从 A 拷贝一次

🐛 问题根因

旧实例（run 在代码改动之前）

字段值来源
name"review-fix-loop"run() 时设置
descriptionundefinedrun() 时没有这个赋值
worker"/path/to/script.js"run() 时设置

新实例（run 在代码改动之后）

字段值来源
name"review-fix-loop"run() 时设置
description"Review-fix loop..."run() 时从 A 拷贝
worker"/path/to/script.js"run() 时设置
关键问题：TUI 读的是 instance.description（数据源 B），而不是从 instance.worker 路径回去读脚本的 meta.description（数据源 A）。

为什么不读 A？ 因为 TUI 只拿到 orchestrator.getInstance(runId)，没有拿到 config-loader。

本质问题：信息不应该是"拷贝"，应该是"引用"。description 是脚本的属性，不是执行的属性。

❌ 当前架构（拷贝模式）

脚本 meta.description
       │
       ▼  run() 时拷贝一次
Instance.description  ← 旧实例没有
       │
       ▼
TUI reads instance.description

问题：旧实例、序列化/反序列化、description 更新都不会同步

✅ 正确架构（引用模式）

脚本 meta.description  ← 始终可用
       │
       ▼  TUI 需要时读取
TUI reads via instance.worker
       → extractMetaViaRegex(worker)
       → meta.description

优点：description 始终从脚本读取，永远是最新的

💡 修复方案

方案 A：TUI 层 fallback（最小改动） 

// WorkflowsView.ts renderView()
let desc = instance.description;
// 如果 instance 没有 description，从脚本 meta 读取
if (!desc && instance.worker) {
  try {
    const meta = await extractMetaViaRegex(instance.worker);
    desc = meta.success ? meta.meta.description : undefined;
  } catch { /* ignore */ }
}

方案 B：orchestrator 层修复（根本修复） 

// orchestrator.ts getInstance()
getInstance(runId: string): WorkflowInstance | undefined {
  const inst = this.instances.get(runId);
  if (inst && !inst.description && inst.worker) {
    // 懒加载：从脚本 meta 回填 description
    const meta = extractMetaViaRegex(inst.worker);
    // extractMetaViaRegex 是 async...需要改签名
  }
  return inst;
}

方案 C：run() 时兜底（最简单） 

// orchestrator.ts run() — 已有 instance.description = workflow.description
// 但 getWorkflow() 可能返回空 description
// 修复：确保 getWorkflow() 始终返回 description
instance.description = workflow.description || "";
