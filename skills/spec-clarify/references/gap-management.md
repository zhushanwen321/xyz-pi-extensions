# Gap 管理：发现、分类、解决

Gap 是追踪循环的燃料。追踪卡住的地方就是遗漏。每个 gap 都是避免后续返工的机会——宁可多发现 10 个不必要的，也不要漏掉 1 个关键的。

## Gap 分类

| 类型 | 含义 | 来源 | 解决方式 |
|------|------|------|---------|
| **F** (Fact) | 代码里有但初稿没提到的信息 | 追踪时发现代码有相关信息 | **主 agent 二次确认** |
| **K** (Knowledge) | 只有用户知道的业务规则 | 追踪卡住且代码无答案 | 直接问用户 |
| **D** (Decision) | 需要做选择的权衡点 | 多种方案都可以，不确定选哪个 | 给方案对比问用户 |

## F 类二次确认（关键设计）

F 是客观事实（代码里有但 spec 初稿没提到的信息），subagent 可能误报：
- 看错代码
- 旧代码已废弃（`@deprecated`）
- 对代码理解有偏差
- 引用的接口/字段实际不相关

主 agent 拿着对话上下文，对"这个事实到底成不成立"有判断力：

| 主 agent 确认结果 | 处理 |
|------------------|------|
| 两边都确认（事实成立且相关）| 转为具体问题问用户 |
| 主 agent 否定（废弃/不相关/理解偏差）| **丢弃，不问用户** |

**二次确认的价值：** 过滤误报，避免基于废弃代码的错误提问打扰用户。

### K/D 不需要二次确认

K/D 是主观判断：
- K（"退款怎么处理"）——主 agent 否定不了（它确实不知道），二次确认只是把 subagent 的发现原样转述，无增值
- D（"用 debounce 还是 idempotency"）——选择权在用户，主 agent 提供选项即可

所以 K/D 直接问用户，不做二次确认。

## Gap 处理（主 agent 回到交互上下文后）

按类型处理：

### F 类型（事实）— 二次确认

对每个 F gap：
1. 主 agent 查看相关代码，判断事实是否成立
2. 成立 + 相关 → 转为具体问题问用户
3. 不成立 / 已废弃 / 不相关 → 丢弃，记录丢弃理由到 tracing-round-{N}.md

**不要把未确认的 F gap 直接问用户。** 先确认。

### K 类型（知识）— 直接问用户

生成具体的、有上下文的问题。

**不要问：** "退款怎么处理？"

**要问：** "代码里有 refund 表但状态只有 pending/completed——退款是部分退款还是全额退款？退款后订单状态变成什么？"

### D 类型（决策）— 给方案对比

提供 2-3 选项 + trade-off，让用户选择。记录决策 + 推理过程到 clarification.md。

### 用户回答后

1. 更新 spec 初稿和 clarification.md
2. 进入 Step 5 收敛复核（派独立 subagent 重新追踪）
3. 收敛复核可能发现新 gap → 回到 gap 处理

## 卡住信号（subagent 追踪时）

以下情况说明遇到了 gap：

1. **"我不知道"** → 需要信息才能继续追踪
2. **"大概是..."** → 在猜测，不是在追踪
3. **"应该可以..."** → 在假设，不是在确认
4. **"这不重要"** → 可能重要，记录为 gap 让主 agent 判断
5. **"这取决于..."** → 有未做的决策（D 类）
6. **"让我看看代码"** → F 类 gap，需要扫描源码

## Stagnation 判定

连续 3 轮追踪 gap 数量不降（新发现 ≥ 已解决），强制收敛。未解决的 gap 标记 `[UNRESOLVED]`，在 spec 中标注交给后续 review 处理。

Stagnation 同时是"需求可能过大"的早期信号——此时提示用户考虑拆分（但拆分机制不在本 skill 范围）。
