## 目标

把 `extensions/subagent-workflow/examples/` 下 4 个"不能直接运行的参考模板"重写为**可被 `workflow run <name>` 直接发现并执行的通用 subagent 编排 workflow**，替代用户日常需要手写复杂 subagent 编排的场景。这是对 ADR-030 决策 3 + D-031 的产品方向调整：模板不再是"复制修改的参考实现"，而是"开箱即用的通用编排工具"。

## 改动范围

### 1. 目录重命名 + 文件改名（语义归位）

`examples/` → `workflows/`，文件 `.example.js` → `.js`：

```
extensions/subagent-workflow/
- examples/chain.example.js        → workflows/chain.js
- examples/parallel.example.js     → workflows/parallel.js
- examples/scatter-gather.example.js → workflows/scatter-gather.js
- examples/map-reduce.example.js   → workflows/map-reduce.js
- examples/README.md               → workflows/README.md（重写）
```

**为什么用 `workflows/` 约定目录而不是 `examples/` + manifest**：
- `workflows/` 是 discovery 的约定目录（`processPackage` 无 manifest 时 fallback 扫 `pkgDir/workflows/`），零 manifest 改动
- 语义正确：这些是 workflow 不是 example
- `meta.name` 与文件名 stem 一致（`chain.js` → stem `chain` = `meta.name`），避免 stem/name 不一致的去重歧义

**package.json 改动**：
- `files` 数组：`"examples/"` → `"workflows/"`
- `pi` manifest：**不变**（约定目录不需要 `pi.workflows` 声明）

### 2. 重写 4 个脚本（agent() 自包含，保留必需参数）

每个脚本用 `agent()` 直接调 LLM，不再依赖不存在的子 workflow。保留 `$ARGS` 必需参数（用户选择），但参数语义从"文件路径"改为"任务描述文本"——更贴合通用 subagent 编排用途。

#### `workflows/chain.js` — 顺序多步处理
**模式**：agent₁ 分析 → agent₂ 变换 → agent₃ 总结，每步输出作下步输入
**必需参数**：`$ARGS.task`（要处理的任务描述字符串）
**流程**：
1. `phase("analyze")` → `agent({ prompt: 分析 task, schema: { insights, keyPoints } })`
2. `phase("transform")` → `agent({ prompt: 基于 insights 产出方案, schema: { plan, actions[] } })`
3. `phase("synthesize")` → `agent({ prompt: 综合 plan 输出最终结论, schema: { summary, recommendation } })`
**返回**：`{ status, phases_run, final: { summary, recommendation } }`

#### `workflows/parallel.js` — 并行多视角分析
**模式**：N 个 agent 从不同角度并行分析同一目标，汇总结果
**必需参数**：`$ARGS.target`（分析目标描述）
**可选参数**：`$ARGS.perspectives`（视角数组，默认 `["security","performance","maintainability"]`）
**流程**：
1. `phase("parallel-analyze")` → `parallel([agent(视角1), agent(视角2), agent(视角3)])`，每个 agent 返回 `{ perspective, score, findings[] }`
2. 汇总：`phase("aggregate")` → `agent({ prompt: 综合 N 个视角的 findings, schema: { overallScore, topIssues[], consensus } })`
**返回**：`{ status, perspectives_analyzed, per_perspective: [...], aggregate: {...} }`

#### `workflows/scatter-gather.js` — 分发-收集
**模式**：一个 agent 把大任务拆成子任务 → 并行处理每个子任务 → 汇总合并
**必需参数**：`$ARGS.task`（要处理的大任务描述）
**流程**：
1. `phase("scatter")` → `agent({ prompt: 把 task 拆成 2-4 个可并行的子任务, schema: { subtasks[{ name, description }] } })`
2. `phase("process")` → `parallel(subtasks.map(s => agent({ prompt: 处理 s, schema: { subtask, result } })))`
3. `phase("gather")` → `agent({ prompt: 合并所有子任务结果, schema: { mergedResult, completeness } })`
**返回**：`{ status, subtasks_total, subtasks_processed, gathered: { mergedResult, completeness } }`

#### `workflows/map-reduce.js` — 映射-归约
**模式**：对 items 数组并行变换 → 归约成单一结果
**必需参数**：`$ARGS.items`（JSON 字符串数组，或 `$ARGS.itemsJson` 文件路径二选一）
**必需参数**：`$ARGS.operation`（map 阶段对每个 item 做什么，自然语言描述）
**流程**：
1. 解析 items（支持 `$ARGS.items` 直接数组 或 `$ARGS.itemsJson` 读 JSON 文件）
2. `phase("map")` → `parallel(items.map(item => agent({ prompt: 对 item 执行 operation, schema: { item, mapped } })))`
3. `phase("reduce")` → `agent({ prompt: 把 mapped[] 归约成最终结果, schema: { reduced, stats } })`
**返回**：`{ status, items_total, items_mapped, reduced: { reduced, stats } }`

### 3. 每个 script 都遵守的约束（lintScript 合规）

- ✅ 含 `agent(`/`parallel(` 入口（4 个都有）
- ✅ top-level await，无 bare IIFE
- ✅ 不用 `result` 作变量名（用 `analyze`/`transform`/`synthesize`/`finding` 等语义名）
- ✅ 用 `schema` 不用 `outputSchema`
- ✅ try-catch 包裹，catch 返回 `{ status: "error", phase, error, message }` 不 crash
- ✅ 缺必需参数在 try 外 throw（清晰错误）
- ✅ `meta.name` = 文件名 stem
- ✅ `description` 用 kebab-case

## 关键设计取舍说明

**为什么用 `agent()` 而不是 `workflow()` 嵌套**：用户明确说"替代日常手写 subagent 编排"。`agent()` 直接调 LLM 是真正的通用工具——用户传 task 描述就能跑。`workflow()` 嵌套需要预定义子 workflow，违背"通用"目标。教学价值由 SKILL.md 承担。

**为什么保留必需参数**：用户明确选择"保留必需参数"。参数是 task/target/items 描述，不是文件路径（chain/parallel/scatter-gather），只有 map-reduce 支持文件路径（itemsJson）作为 items 的可选输入源。这让脚本贴近真实使用而非 demo。

**为什么不加 ResourceSource 类型区分"内置"**：当前 workflow 域 `source` 只有 `saved`/`tmp` 两值，加 `bundled` 需要改 5+ 处类型映射，收益（UI 能标"内置"）不抵改动成本。`workflow list` 会显示 `[saved] chain` 等同于用户脚本，用户可通过 description 里的"通用编排"字样区分。
