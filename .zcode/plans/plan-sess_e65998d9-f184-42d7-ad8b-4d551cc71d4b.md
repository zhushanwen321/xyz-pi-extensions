## PoC 目标

验证：background subagent 执行期间，扩展层能否把 text_delta 经时间窗合并后，通过 `ctx.ui.setWidget` 通道稳定传输到 RPC stdout（每秒 5-10 条持续负载）。

## PoC 改动链路（4 个文件）

### 1. `session-runner.ts` — hook text_delta

在 `RunOptions`（L165-195）加一个可选回调字段：
```ts
onStreamDelta?: (delta: string) => void;
```

在 `agentEvent` 出口函数（L474-478）加一行：
```ts
if (event.type === "text_delta") opts.onStreamDelta?.(event.delta);
```

这是单点改动，text_delta 在到达 `opts.onEvent?.(event)`（对 background 是 no-op）之前分流到新回调。

### 2. `subagent-service.ts` — 合并 + 转发

**新增字段**：
```ts
private streamSink: StreamSink | null = null;
```
其中 `StreamSink` 是最小接口 `{ setWidget(key: string, lines: string[] | undefined): void }`。

**initSession 签名扩展**：接收 `ui` 参数（从 index.ts session_start 注入 ctx.ui 的最小子集）。

**thread onStreamDelta**：
- `runAndFinalize`（L563）接收 `onStreamDelta` 参数，透传到 `runSpawn` 的 opts
- `kickOffBackground`（L646）调 `runAndFinalize` 时构造 delta 合并闭包：
  - 按 `record.id` 维护一个 text buffer（`string`）
  - 时间窗合并：100ms 内的 delta 追加到 buffer，timer 到期时调 `streamSink.setWidget("subagent-stream-<id>", buffer.split("\n"))`
  - leading edge：第一个 delta 立即触发一次 setWidget（让前端尽快看到开始）
  - trailing edge：timer 到期时刷新最终累积

**终态清理**：在 `finalizeRecord`（L729）或 `runAndFinalize` 的 finally 中调 `streamSink.setWidget(key, undefined)` 清除 widget。

### 3. `index.ts` — 注入 ctx.ui

在 `session_start` handler（L198-218）中，把 `ctx.ui` 的 `setWidget` 方法包装成 `StreamSink`，传给 `service.initSession`：
```ts
service.initSession({
  pi,
  sessionId,
  streamSink: { setWidget: (key, lines) => ctx.ui.setWidget(key, lines) },
});
```

注意：只绑 setWidget 方法（不持有整个 ctx），避免 ctx 生命周期问题。`ctx.ui` 实例在整个 session 期间由 mode 持有，setWidget 方法引用稳定。

### 4. xyz-agent runtime（只读验证，不改代码）

PoC 阶段不实现 runtime 侧的 subagent streaming 消费。验证方式：

**方式 A**（原始 stdout 观测）：启动 pi RPC 模式，触发 background subagent，直接看 RPC stdout 输出——确认是否有 `extension_ui_request { method: "setWidget", widgetKey: "subagent-stream-*" }` 帧到达，频率是否在预期范围（5-10 Hz）。

**方式 B**（xyz-agent runtime 日志）：在 xyz-agent 的 EventAdapter 里临时加一行 console.log 观察 setWidget 事件到达情况。这是临时调试代码，PoC 后删除。

## PoC 验证清单

| 验证项 | 方法 | 预期 |
|--------|------|------|
| text_delta 被 hook | session-runner 加临时 log | 每个 token delta 触发 onStreamDelta |
| 合并窗口生效 | 观察 setWidget 频率 | 5-10 Hz 而非每 token 一条 |
| RPC stdout 到达 | 看 RPC stdout 原始输出 | 有 subagent-stream-* widget 帧 |
| 主 session 不受影响 | 观察 agent 正常交互 | stdout 无堵塞、无延迟 |
| 终态清理 | subagent 完成后 | widget 被清除（setWidget key, undefined）|
| 内存稳定 | subagent 生成长回复（2000+ token）| buffer 不无限增长，stdout 不堆积 |

## 不做的事（PoC 范围外）

- 不改 xyz-agent runtime 的 EventAdapter/EventInterpreter（不加 subagent.* 事件路由）
- 不改 xyz-agent 前端（不接 setWidget → chatStore）
- 不实现 thinking_delta 转发（只验证 text_delta）
- 不实现 workflow agent call 的 streaming（只验证 background subagent）
- 不加正式测试（PoC 验证后决定是否正式实现再加测试）

## PoC 后的决策点

如果验证通过（setWidget 通道稳定）→ 设计 A+C 共存的正式实现方案
如果验证失败（性能问题/事件丢失）→ 回退到路径 C（fs.watch JSONL，message 级刷新）
