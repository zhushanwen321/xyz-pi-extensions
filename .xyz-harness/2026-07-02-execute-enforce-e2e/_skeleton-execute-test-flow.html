TL;DR核心结论

  • 不用 workflow，用 tool。tool 能管状态（闭包 state）+ 强制参数（typebox schema）+ 拒收非法操作（throw），恰好满足「状态计算 + 上报校验」需求。
  • tool 全局可见，但谎报过不了。「tool 只在 workflow 暴露」不可行（pi extension tool 是进程级注册，无 scoped 机制）。但不需要隐藏——complete 的机器校验焊死后，主 agent 传假证据直接 throw 拒收。
  • 状态计算：AI 只传证据，status 机器重算。complete(caseId, actual, screenshotPath) → 机器校验截图存在 + 拿 expected 逐字段比对 → 重算 pass/fail。AI 填的 status 丢弃。
  • 两道焊死的关：① complete 运行时 requireParam 校验 actual+screenshotPath 必填（schema Optional 因同一 schema 服务 4 action，typebox 不支持条件必填），缺则 throw；② get-result 全覆盖拒绝，未完成 throw 阻塞 goal complete。
  • soft 隔离可选：sessionId 机制防误用/干扰，但不防恶意（真防恶意靠机器校验）。

为何用 tool 不用 workflow

维度workflow 方案tool 方案
状态管理fs 读写 json（双方都碰）闭包 state（单点管理，todo 模式）
参数强制无（prompt 要求）typebox schema 层拒收残缺
谎报防护靠 json 格式约定complete schema + 机器重算 status
全覆盖校验脚本 fs 读 json 自查get-result throw 拒绝
跨进程限制worker 线程不能调 tool无（tool 是 pi 原生机制）
驱动循环机器 JS（物理不可跳）AI prompt（可跳，收尾门兜底）
实现成本高（跨包改 workflow）中（单 tool，复用 todo 模式）

核心取舍：tool 放弃了 workflow 的「物理不可跳」（AI 可选择不跑），换来更强的「上报必须真实」（schema + 机器重算）。上报真实性比驱动不可跳更重要——即使 AI 跳了，收尾门（get-result throw）兜底，goal complete 不了；但 AI 跑了谎报，tool 方案下过不了机器校验，workflow 方案下 json 自由格式可能漏。

四个 action 详解

① init — 主 agent 调一次

init(planPath: string) → { sessionId, caseCount, layers }
  • 读 plan.md，机器解析 mock 命令 + real 用例清单（ID/预期/步骤/fixture）
  • 校验 plan 格式（复用 check_plan.py 逻辑：测试层、expected 具体值、real ≥1）
  • 生成 sessionId（uuidv7），存进闭包 state
  • 格式不合法 → throw（plan 设计阶段就该过 check_plan.py，这里是兜底）

② get — subagent 循环调

get(sessionId: string) → { caseId, layer, expected, steps, fixture } | null
  • 返回下一个 status=pending 的用例，标记 in-progress（防多 subagent 重复认领）
  • 返回完整上下文：expected（机器用于后续重算）+ steps + fixture 位置
  • 无 pending → null（信号：该层跑完，可进收尾门）
  • sessionId 不匹配 → throw

③ complete — subagent 跑完调（核心）

complete(sessionId, caseId, actual: {url?, text?, domAttr?...}, screenshotPath: string) → { machineStatus, verifiedAt }
  • 🔒运行时 requireParam 校验：actual/screenshotPath/caseId 必填（schema Optional 因同一 schema 服务 4 action 不支持条件必填，运行时 throw 兑现）。缺任一 → tool 层 throw 拒收（AI 拿不到「上报成功」）
  • 🔒机器校验证据：fs.existsSync(screenshotPath) → 不存在 throw
  • 🔒机器重算 status：拿 init 缓存的 expected，与 actual 逐字段比对 → pass/fail。AI 传的 status 参数（若有）丢弃
  • 存状态：{ caseId, machineStatus, actual, screenshotPath, verifiedAt }
  • 返回 machineStatus（AI 拿到机器判定，可决定是否重试）

④ get-result — 收尾门调

get-result(sessionId) → { allPassed, cases: [{id, status, evidence}], coverage } | throw
  • 校验全用例 completed（无 pending / in-progress 遗留）→ 否则 throw
  • 校验全 pass → 否则 throw
  • 返回聚合结果（每条用例的机器判定 status + 证据路径）
  • 这是收尾门的硬阻塞：goal_control(complete) 前必须先调 get-result，throw 则 goal complete 失败

状态计算（核心机制）

一句话：AI 调 complete 时只传「证据」，status 由机器重算，AI 填的丢弃。

为何这样设计

  • AI 自报 status 不可信：AI 可能跑都没跑，直接传 status: pass。若采信，谎报成功。
  • AI 传实际观测值 actual 可信（有证据约束）：actual 是 AI browser skill 抓到的真实 DOM/URL/text。配合 screenshotPath 存在性校验，AI 必须真跑了浏览器才能产出这些。
  • expected 来自 plan（机器解析缓存）：check_plan.py 已强制 plan 的 expected 是具体值（如 url === '/profile'），不是模糊描述。机器拿这个具体值比对。
  • 比对在 tool 内部（机器 JS）：AI 摸不到比对逻辑。AI 只能传 actual，pass/fail 是机器算的。

状态生命周期

status何时设置谁设
pendinginit 时（plan 解析）机器
in-progressget 返回时机器
passed / failedcomplete 时（expected vs actual 比对）机器（AI 传的丢弃）

两道焊死的关

关 1complete 的 schema Optional + 运行时 requireParam + 机器校验

防线 1typebox schema Optional（同一 schema 服务 4 action，不支持条件必填）+ 运行时 requireParam throw：actual/screenshotPath/caseId 缺则 throw
防线 2机器 fs.existsSync(screenshotPath)：AI 没真跑浏览器就没截图 → throw
防线 3机器重算 status：expected（plan 缓存）vs actual（AI 传）逐字段比对。AI 填的 status 丢弃

三层叠加：AI 必须真跑、必须截图、必须实际值对得上 plan 预期。任一缺失过不了。

校正措辞（v4 实现对齐）：非「schema 层拒收」，是「schema Optional + 运行时 requireParam throw」。效果一致（都 throw），但机制需准确——条件必填靠运行时校验（见 AGENTS.md「Tool 设计」）。

关 2get-result 的全覆盖拒绝

触发goal_control(complete) 前主 agent 必调
校验全用例 completed + 全 pass，否则 throw
效果AI 跳过任一用例没调 complete → 该用例仍 pending → get-result throw → goal complete 失败

软隔离（可选，不防恶意）

为何「不隐藏」仍安全：tool 全局可见，但主 agent 调 complete 传假证据 → 机器校验 throw；调 get-result 提前放行 → 全覆盖校验 throw。机器校验让全局可见变得无害。

若仍想防误用/干扰（非恶意），加 sessionId 机制（照搬 todo 闭包 state）：

  • init 创建 session，返回 sessionId；后续 action 必须带 sessionId
  • 主 agent 不跑 workflow 就拿不到 sessionId → 调啥都 throw
  • 局限：主 agent 能自己调 init 拿 sessionId。真防恶意靠机器校验（关 1/关 2），不靠 sessionId

强制力对照表

核心：标「机器」的环节 AI 完全碰不到。tool 把「上报真实性」和「全覆盖」两道关焊死。

环节执行者AI 能否干预强制机制
plan 解析 + expected 缓存机器否init 时 fs 读 + 正则解析
用例分配（get）机器否闭包 state 遍历，标记 in-progress
real 用例执行AI是subagent + browser skill
证据上报（complete）AI是（受限）schema Optional + 运行时 requireParam 强制 actual+screenshot，缺则 throw
status 判定机器否expected vs actual 逐字段重算，AI 填的丢弃
全覆盖校验（get-result）机器否有 pending → throw 阻塞 goal complete
驱动循环（AI 主动去跑）AI是⚠ 可跳过，收尾门兜底

v4 确定方向：纯 tool 驱动（get/complete/get-result），不隐藏，靠 schema + 机器重算 status + 全覆盖 throw 三重校验。可开始落地 tool 扩展。
