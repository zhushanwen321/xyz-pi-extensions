1当前实现：纯 In-Process 执行

所有 subagent 在同一个 Node.js 进程内执行，通过 Pi SDK 的 createAgentSession() 创建独立 session。已落地 代码中没有任何 spawn / child_process 进程隔离后端，isolation 字段仅支持 undefined 和 "worktree"。

4适用场景

当前 In-Process 模式

✅Coding 主力任务
代码生成、重构、修 bug —— 高频、短生命周期、需完整工具集与快速响应。
→ 完美匹配 · 默认模式
✅工具密集型编排
需要 fork 主 session 上下文、访问全部 extension tools、嵌套调用其他 subagent。
→ 完美匹配 · 进程内天然支持
✅需 steer / abort
运行中动态注入消息（session.steer()）或优雅终止（session.abort()）。
→ 完美匹配 · 进程内直调
✅短时后台（等几分钟拿结果）
wait:false + backgroundId 轮询 —— 当前 background 模式的典型用法。
→ 匹配 · L1/L2 持久化已覆盖
⚠️git worktree 隔离执行
需文件系统级隔离（并行 branch 实验）—— isolation:"worktree" 已支持。
→ 匹配 · 但仍是同进程，OOM 会传染
❌跨重启的长跑任务（L3）
启动 background agent → 关 Pi → 重启 → 期望它还在跑。当前架构做不到。
→ 不支持 · 是 spawn 的唯一不可替代价值
❌不可信代码沙箱
执行来源不明的代码，需 OS 级进程隔离防止崩溃/OOM 拖垮主进程。
→ 不支持 · in-process 下 OOM 会杀主进程
⚠️超长索引 / 批量处理
索引整个 monorepo、批量重构 —— 可能跑数十分钟。建议用 schedule_prompt（已有工具）而非 subagent。
→ 存疑 · 是否该由 subagent 承担？

Spawn 模式（若实现）能补的缺口

🔮跨重启持久任务（L3）
detached 子进程独立于主 Pi 存活，重启后通过 pid 探活 + result.jsonl 恢复状态。
→ spawn 的核心价值 · 需先确认是否真实高频
🔮进程级崩溃隔离
不可信代码在独立进程跑，OOM/crash 不传染主 Pi。
→ 可用 worktree + maxTurns 替代实现"防失控"

5抉择的核心：一个问题

技术对比已无悬念 —— 两者都可行，混合架构也设计好了。 真正的抉择点不是技术问题，是需求频率问题：
"跨进程恢复运行中的 subagent（L3）"是不是真实的高频痛点？

抉择前必须验证的事实（避免基于假设决策）
  1. 实际使用数据：~/.pi/agent/subagents/*/history.jsonl 中 background 记录比例与时长分布 —— 若大多几分钟内完成，L3 需求弱。
  2. 是否该用 schedule_prompt：跨重启的持久任务是否本就该用已有的 schedule_prompt 工具，而非 subagent 承担？
  3. ADR-022 约束是否仍成立：当时的迁移理由（性能、工具完整性）现在是否还成立？Pi SDK 是否有新变化？
  4. 第三方包状态：tintinweb 版 pi-subagents 的 async-execution.ts 是否已卸载 —— 若还在，可参考其 spawn 模块而非从零实现。
