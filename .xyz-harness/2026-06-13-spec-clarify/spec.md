# Spec-Clarify — 解决澄清遗漏

## Background

当前 brainstorming 环节的核心痛点：**交互式提问澄清看似好用，实际不能覆盖所有需要澄清的环节，且遗漏的是比较重要的功能。** 即便简单需求也存在澄清不足。

本次 spec **只解决这一个核心问题**（下称「问题 1」）。规模较大的需求覆盖问题（「问题 2」）通过递归澄清解决，思路清晰，作为后续独立 spec 处理，本次不涉及。

### 遗漏的根因

现有提问机制有三个叠加的缺陷：

1. **提问靠灵感，不靠清单。** AI 听到需求→想到什么问什么→觉得差不多了就停。没有任何外部参照系提醒它"你还该看看退款这块"。结果：同一需求换时机/换对话，问的维度可能完全不同，质量下限无法保证。

2. **停止条件依赖自我判断，而 AI 的盲区恰恰是"不知道自己不知道"。** 现有停止条件是"能完整描述方案而不猜测"——AI 觉得自己能描述了，其实遗漏了一整块。

3. **后置的完整性检查救不了前置的遗漏。** Six-Element Check 是形式合规（框填没填），不查内容完整；Ambiguity Marking 只能标记已写出的模糊词，对根本没想到要写的东西无能为力。

**根因总结**：遗漏不是因为 AI 笨或问得不够多，是因为提问的驱动力是"AI 的灵感"而非"外部强制清单"，且没有任何机制能在 AI 不知道该问什么时主动暴露盲区。

### 核心认知：两条执行上下文，职责分离

整个解法基于一个判断：**AI 自己查不出自己的盲区。**

因此必须把"追踪遗漏"从主 agent 手里拿出来，交给**隔离上下文的独立 subagent**。两个上下文各司其职：

| | 主 agent（交互上下文） | 独立 subagent（隔离上下文） |
|---|---|---|
| **职责** | 交互：提问、给方案、写初稿 | 追踪：5 视角强制枚举，找 gap |
| **输入** | 用户对话 + 项目代码 | 只读：需求文本 + spec 初稿 + 源码 |
| **关键特征** | 带着完整对话上下文 | **不知道主 agent 聊了什么** |
| **为什么这样分** | 对话需要上下文才能聊清楚 | 追踪必须隔离，否则继承主 agent 的确认偏误 |

**独立 subagent 是唯一的正式追踪者。主 agent 不做系统追踪。** 这是整个设计的基石。主 agent 如果自己做追踪，会带着"我已经问过这块了"的预设去走，对预设一致的部分快速跳过——这正是盲区产生的地方。

## 核心机制

### 机制 1：5 视角场景追踪（forcing function）

解决"不知道该问什么"——给 AI 一个有限清单，必须逐项核对。5 个视角覆盖软件需求最核心的正交切面，每个视角有强制检查项：

| 视角 | 强制回答的问题 | 暴露的典型遗漏 |
|------|--------------|--------------|
| **User Journey** | 每个操作：成功后下一步？中途放弃？重复操作？权限不足？超时？ | 确认缺失、中间状态、并发、权限边界 |
| **Data Lifecycle** | 每个实体：创建校验？唯一约束？删除后引用数据怎么办？软删/硬删？ | 级联删除、唯一性边界、数据迁移 |
| **API Contract** | 每个接口：所有错误码？幂等？输入边界值？分页上限？ | 只定义 happy path、缺错误码 |
| **State Machine** | 每个状态机：非法转换怎么处理？僵尸状态？状态可见性？ | 回滚缺失、非法转换提示 |
| **Failure Path** | 每个操作：超时？部分失败？重试策略？事务一致性？ | 补偿机制、降级方案 |

**视角是框架，不是工作量。** "给订单列表加导出按钮"这种简单需求，走 5 视角可能 10-15 分钟——但能抓到"导出范围""导出格式""失败处理"这几个真实遗漏点。重不重取决于需求本身，不取决于框架。

**视角支持降级，但不允许无声跳过。** 不适用视角必须写明降级理由（"本需求不变更状态机，不追踪"+ 依据）。不写理由直接跳过 = 逃避追踪，违背 forcing function 原则。这是防止 AI 偷懒的关键约束。

**为什么是这 5 个视角：** 它们正交覆盖了需求最容易遗漏的几个切面——用户行为、数据流转、接口契约、状态合法、失败路径。少于 5 个会有盲区，多于 5 个边际收益递减且增加 AI 负担。具体的追踪模板放在 skill 的 references 中，不进本 spec。

### 机制 2：gap 分类（决定解决方式）

追踪卡住的地方就是 gap。gap 分三类，直接决定"该问用户还是该自己查代码"：

| 类型 | 含义 | 解决方式 | 为什么要分类 |
|------|------|---------|------------|
| **F（Fact）** | 代码里有但主 agent 还没读到的信息 | subagent 扫代码验证 → **主 agent 二次确认** → 两边都认才问用户；主 agent 否定则丢弃 | 客观事实可查，二次确认过滤误报 |
| **K（Knowledge）** | 只有用户知道的业务规则 | 直接问用户（具体、有上下文的问题） | 主观判断，没客观参照，直接问 |
| **D（Decision）** | 需要做选择的权衡点 | 给用户 2-3 方案对比，记录决策+推理 | 选择权在用户，AI 只提供选项 |

**F 类需要二次确认，K/D 不需要——这是关键设计决策（D-1）。**

F 是客观事实，subagent 可能误报（比如它看到的旧代码已废弃，或它对代码理解有偏差）。主 agent 拿着对话上下文，对"这个事实到底成不成立"有判断力：两边都确认才值得打扰用户；主 agent 否定就丢弃，不浪费用户时间。

K/D 是主观判断，二次确认没意义——"退款怎么处理"这种问题，主 agent 否定不了（它确实不知道），只会把 subagent 的发现原样转述。所以直接问用户。

### 机制 3：收敛循环（独立复核）

主 agent 对自己构建的理解有确认偏误。解决方式：**每轮 gap 处理完后，再派一次独立 subagent 重新追踪**（同样隔离上下文）。它不知道上一轮查过什么，从零审视，反而能发现主 agent 的盲区。

**停止条件：独立 subagent 跑完 5 视角，无新 gap，即收敛。** 不依赖 AI 的自我判断（根因 2），而是靠外部独立验证给出收敛信号。

**Stagnation 保底：** 连续 N 轮（建议 3 轮）gap 数量不降（新发现 ≥ 已解决），强制收敛。未解决 gap 标记 `[UNRESOLVED]` 留给后续 review 处理。防止需求本身过大导致无限循环——这同时是"问题 2"的早期信号（此时应建议拆分，但拆分机制不在本 spec 范围）。

## 流程

```
[主 agent: 交互上下文]
  Step 1: Quick Overview          → 浏览项目，< 30s
  Step 2: 基于理解的交互提问        → 逐个提问（ask_user），写 spec 初稿 + 轻量 clarification.md
                                    （刻意不做 5 视角追踪 — 追踪交给隔离 subagent）
  ↓ 产出：spec 初稿 + clarification.md

[独立 subagent: 隔离上下文，只读需求+初稿+代码]
  Step 3: 5 视角强制追踪           → 独立重跑，卡住点 = gap，分类返回（F/K/D）
  ↓ 产出：gap 列表

[主 agent: 回到交互上下文]
  Step 4: gap 分流处理
    F 类 → 二次确认：两边都认→问用户；主 agent 否定→丢弃
    K 类 → 直接问用户
    D 类 → 给方案对比
    处理完 → 更新 spec / clarification.md

  Step 5: 收敛判定
    再派独立 subagent 追踪
    无新 gap → 收敛，进入 Step 6
    有新 gap → 回 Step 4（Stagnation 保底：连续 N 轮强制收）

  Step 6: spec 定稿                → 整理 frontmatter，调用 gate 进入后续 review
```

**循环的粒度：** 每次"再追踪"是完整重跑 5 视角，不是增量。因为独立 subagent 不知道上轮结果，无法做增量——它从零审视正是其价值所在。这会带来重复追踪的成本，但这是换取"无盲区"的必要代价。简单需求通常 1-2 轮就收敛，成本可控。

## 关键约束

### C-1: 主 agent 不做系统追踪

主 agent 在 Step 2 只做对话式澄清，**不主动走 5 视角**。这是设计的基石——主 agent 带着对话上下文，对"已经讨论过"的部分会有预设性快速跳过，这正是盲区产生的地方。追踪必须由隔离上下文的 subagent 做。

主 agent 在 Step 2 做的澄清是"对话能聊清楚的聊清楚"，不是"系统穷举所有切面"。两者职责不同。

### C-2: 独立 subagent 必须隔离上下文

subagent 不能继承主 agent 的对话历史。它的输入只有：需求文本、spec 初稿、clarification.md、项目源码。这保证它从零审视，不继承主 agent 的预设和确认偏误。

### C-3: 5 视角是 forcing function，不是建议

每个视角必须逐一核对，不适用必须写降级理由。这是"外部清单"的核心——强制 AI 对每一项给出回答，回答不上来就是 gap。如果允许 AI 自由选择"我觉得重要的视角"，就回到了根因 1（靠灵感）。

### C-4: F 类必须二次确认

F 类 gap 不能直接问用户。必须先经主 agent 二次确认——主 agent 拿着对话上下文判断这个事实是否成立。两边都认才问用户，主 agent 否定则丢弃。这过滤了 subagent 的误报，避免浪费用户时间。

K/D 不需要二次确认，直接问用户（见 D-1）。

### C-5: clarification.md 是轻量的

不强制五维度表格（Entities/Operations/State Machines/Actors/Constraints）。它只是记录已知信息和需求拆解的载体，具体形式由 skill 决定。5 视角的追踪产出本身就是"模型"，不需要再抽象一层强制结构。

过重的结构会让简单需求写 clarification.md 变成负担，违背"简单需求也适用"的目标。

## 决策记录

### D-1: F 类二次确认，K/D 不确认

**决策：** F 类 gap 必须经主 agent 二次确认（两边都认才问用户，主 agent 否定则丢弃）；K/D 类直接问用户。

**理由：**
- F 是客观事实（"代码里是否有 partial_refund 方法"），有客观参照可查。subagent 可能误报（看错代码、旧代码已废弃）。主 agent 拿着对话上下文能判断这个事实成不成立，过滤误报。
- K/D 是主观判断（"退款怎么处理""用 debounce 还是 idempotency"），没有客观参照。主 agent 否定不了（它确实不知道），二次确认只是把 subagent 的发现原样转述，无增值。直接问用户更高效。

**代价：** F 类多一步主 agent 确认，增加少量延迟。换取：过滤误报，避免 subagent 的理解偏差打扰用户。值得。

### D-2: 追踪完全交给独立 subagent（方式 B）

**决策：** 主 agent 不做系统 5 视角追踪，追踪完全由隔离上下文的 subagent 完成。主 agent 发现遗漏只通过对话自然发生，不形式化。

**替代方案（已否决）：方式 A**——主 agent 自己追踪，subagent 只做复核比对。否决理由：主 agent 带着对话上下文做追踪，会继承"我已经问过这块"的预设，对预设一致部分快速跳过——这正是盲区。方式 B 让追踪者完全隔离，消除了这个偏差。

**代价：** subagent 从零重跑，有重复追踪成本。但简单需求 1-2 轮收敛，成本可控；且这是换取"无盲区"的必要代价。

### D-3: 不引入 complexity-assess / decompose / L1-L2 分级

**决策：** 本 spec 不做复杂度分级，所有需求走同一条 L0 流程。

**理由：** 复杂度分级是为"问题 2"（大需求覆盖）设计的。本次专注问题 1（澄清遗漏）。先把简单问题解决好是解决问题 2 的基础——如果 L0 流程本身仍会遗漏，L1/L2 的递归只是在每一层重复同样的遗漏。L1/L2 机制（manifest、子系统、合约、拓扑排序）是推测性功能，无真实需求验证，按 CLAUDE.md「不加推测性功能」原则砍掉。

### D-4: Stagnation 保底用固定轮数，不做复杂判定

**决策：** 连续 3 轮 gap 不降即强制收敛，不做"内层/外层循环""升级判定"等复杂控制流。

**理由：** 旧归档 spec 设计了 4 条升级判定条件、两层循环嵌套，调试成本极高。这些形式化增加了 AI 的认知负担（纠结"我在哪层""这个 gap 走哪条路"），反而偏离澄清核心目标。固定轮数简单可靠，未解决 gap 标记留给后续。

## 明确不做（防止范围蔓延）

以下内容在本 spec 范围**之外**，明确不做。防止设计再次膨胀：

- **不做复杂度分级（L0/L1/L2）** — 为问题 2 设计，后续独立 spec
- **不做 decompose / 子系统 / manifest.yaml / children/ / api-contracts.md** — 同上
- **不做 complexity-assess 操作/tool** — 同上
- **不做 NEEDS_USER 的复杂升级判定** — gap 处理就是处理，不分"局部 vs 完整"
- **不做两层循环形式化（内层/外层）** — AI 自然迭代，不强制循环语义
- **不做 model_version ≥ 2 等硬性收敛指标** — "无新 gap"即收敛
- **不做 gap 优先级（P0/P1/P2）和 Gap Tracker 表格** — 保留 F/K/D 分类即可
- **不强制 clarification.md 五维度表格** — 轻量记录即可
- **不新建 spec-clarify-phase 的 gate/review-loop 维度配置** — gate 机制由所属 workflow 定义，本 spec 只定义 clarify 本身的流程

## 验收标准

### AC-1: 解决根因 1（提问靠灵感）

- [ ] 主 agent 在 Step 2 不主动走 5 视角（C-1）
- [ ] 存在一个隔离上下文的 subagent 承担追踪职责（C-2）
- [ ] 5 视角作为 forcing function 强制执行，每个视角必须核对或写降级理由（C-3）

### AC-2: 解决根因 2（停止靠自我判断）

- [ ] 收敛由独立 subagent 复核判定，不依赖主 agent 的自我判断（机制 3）
- [ ] 收敛条件是"无新 gap"，不是 checklist 完成数或 model_version

### AC-3: F 类二次确认

- [ ] F 类 gap 经主 agent 确认后才问用户；主 agent 否定则丢弃（C-4, D-1）
- [ ] K/D 类直接问用户，不做二次确认（D-1）

### AC-4: 简单需求也适用

- [ ] "给订单列表加导出按钮"这类简单需求能走完全流程，不因框架过重而负担过大（C-5）
- [ ] 简单需求通常 1-2 轮收敛

### AC-5: 范围控制

- [ ] 未引入 complexity-assess / decompose / manifest 等问题 2 的机制（D-3）
- [ ] 未引入两层循环 / 升级判定 / P0-P2 优先级等过度工程（D-4）

## 业务用例

### UC-1: 简单需求（主场景）

用户说"给订单列表加导出 Excel 按钮" → 主 agent Quick Overview → 对话澄清（导出触发、格式等能聊清的）→ 写 spec 初稿 + 轻量 clarification.md → 派独立 subagent 追踪 → 发现 gap（导出范围？失败处理？大文件？）→ F 类二次确认 / K 类直接问用户 → 更新 spec → 再追踪无新 gap → 收敛 → spec 定稿。

### UC-2: 中等需求

用户说"做一个优惠券系统" → 同样流程，subagent 追踪会发现更多 gap（券状态机、叠加互斥、核销流程、失效回收）→ 多轮收敛 → 定稿。此时若 gap 数量持续不降触发 Stagnation，提示用户需求可能过大，建议拆分（但拆分机制不在本 spec）。

### UC-3: F 类误报被过滤

subagent 报告 F 类 gap："代码里有 `partial_refund` 方法，退款是否支持部分退款？" → 主 agent 二次确认：查看代码发现是已废弃的旧实现 → 否定该 gap → 不问用户，避免打扰。
