§1 系统设计立场

核心计算是什么？ CW 的核心是流程状态机的编排 + gate 验证，不是业务规则编排，也不是技术流程编排（不 spawn 进程跑 agent）。CW 是一个有状态的请求处理器：接收 agent 的 tool call → 校验状态 → 跑 gate → 更新状态文件 → 返回 nextAction。

这个定位决定分层：CW 是无 IO 副作用的编排层（除读写 _cw.json 和调用 check 脚本/git），不承担 agent 执行、UI 渲染、budget 管理（那些归 goal / todo / subagent）。

§2 三层职责

采用简化三层（非 DDD4），因为 CW 的"领域逻辑"就是状态机转换 + gate 调度，无复杂业务规则。

Tool 接口层

index.ts

  • registerCodingWorkflowTool + execute
  • typebox schema 校验入参
  • 按 action 分派到应用层
  • 组装出参 status / gatePassed / nextAction
应用层

action handlers + 状态机

  • 8 个 handler: create / plan / clarify / detail / dev / test / retrospect / closeout
  • 状态机校验（expectedStatus 强制线性）
  • 编排 gate 调用 + 状态流转
  • 渐进式提交的累计判定逻辑
基础设施层

gate 检查器 + 存储 + 解析器

  • GateRunner: subprocess 调 check_*.py
  • GitValidator: git 命令校验 commit
  • JudgeByExpected: test-orchestrator 内化
  • CwStore: 原子读写 _cw.json
  • PlanParser: 解析 plan/clarify/detail JSON
为何不用 DDD4：CW 无领域聚合（topic 是简单状态机封装），无复杂业务规则（状态机本身就是核心约束），无 Port（check 脚本/git 是稳定的命令式调用不需要可替换抽象）。三层够用且不过度设计。

action 入参与实现约定（跨模块共用）

topic 定位

除 create 外，所有 action 入参含 topicId（或 topicDir，CW 内部解析）定位当前 topic。create 返回 topicId，后续 action 用该 topicId。

Pi 事件订阅

CW 是纯 registerTool，不订阅 pi.on 事件。CW 是无状态请求处理器，靠 _cw.json 持久化状态，不依赖事件回调。

错误返回

遵循项目规范，错误用 throw new Error()，不返回伪装成功的 content 文本（错误成功模式）。CW 的 throw 被 Pi 捕获后转述给 agent。

dependsOn / parallelGroup

CW 仅记录这两个字段（从 plan/detail JSON 写入 _cw.json），不消费做依赖编排——依赖编排归 agent + coding-execute skill 的 Wave DAG 派发，CW 只校验 Wave 维度的 commit 完整性。Wave 可乱序提交。

workspacePath 语义

_cw.json.workspacePath 记录 topic 对应项目目录的绝对路径，GitValidator 在该目录下执行 git 命令（支持一个 agent 跨多项目场景，CW 不假设 cwd）。create 时由 agent 传入或默认 cw 进程的 cwd。

§4 状态机（核心不变式）

状态定义

lite: created → planned → developed → tested → retrospected → closed
mid:  created → clarified → detailed → developed → tested → retrospected → closed

状态机表（紧约束，显式转换表）

当前状态合法 action流转后状态额外前置（跨阶段级联）
—（无 topic）create（入口）created无
created (lite)planplanned无
created (mid)clarifyclarified无
clarified (mid)detaildetailed无
planned (lite)devdeveloped无（首次有效提交即流转）
detailed (mid)devdeveloped无（首次有效提交即流转）
developedtesttestedrequire: dev 阶段全 Wave committed
testedretrospectretrospectedrequire: test 阶段全 testCase passed
retrospectedcloseoutclosed无
两重校验（D-009 主强制点）——CW 在 action handler 入口做两道校验：
  • ① 状态机线性：currentState ∈ expectedStatuses[action]，不满足 throw illegal state transition，不跑 gate。
  • ② 跨阶段 gatePassed 级联（渐进式阶段进下一阶段时）：上一阶段累计完成信号必须为 true（dev 全 Wave committed / test 全 case passed），否则 throw previous phase incomplete，不跑 gate。
第二道解决「developed/tested 态内可跳进下一阶段」的漏洞——仅靠 status 相等无法防 agent 丢掉剩余 Wave/case 直接进下一阶段。图中用红色虚线标注此级联边。

渐进式 action 的状态语义

dev 和 test 是渐进式的：进入态在首次有效提交时流转（如 planned→developed）；态内推进不流转状态，更新 _cw.json 的 waves.committed / testCases.status；态完成信号靠 gatePassed: true 表达（全 Wave committed / 全 case passed），不靠状态流转。agent 据 gatePassed 判断能否进下一步，status 表示"当前所处阶段"。

严格度选择：紧（显式转换表）而非松（只守终态）

选紧约束的理由：CW 的核心价值就是强制线性流转。松约束（只检查终态 closed）会让中间状态可乱跳，失去防跳过能力。显式转换表的代价（加 action 要改表）远小于收益（防跳过）。

§5 Gate 注册表（核心配置）

§5.1 gate 强度定义（4 档）

gateTier含义agent 可信度
weak-structural仅检查文件存在 + 结构完整（章节/占位符）低，内容正确性靠 review/人判
medium-gitcommitHash 经 GitValidator 三项校验（存在/属本仓库/非空）中（dev，有真实凭证不重算业务）
medium-coveragecommitHash 校验 + 信自然语言声明，不重算业务断言（mid test）中（信声明）
strong-recompute机器重算 actual vs expected，丢 AI 声明高（密封）
medium 拆分诚实标注：medium-git（dev，commitHash 经 GitValidator 三项校验）与 medium-coverage（mid test，commitHash 校验 + 信声明但不重算业务断言）强度不同，诚实区分。lite test 的 strong-recompute 是唯一密封档（机器重算丢 AI 声明）。

review 桩机制（修正：review 文件由 skill 阶段产出，非 CW 运行时产）：
  1. skill 阶段（mid-plan / mid-detail-plan）的 review-fix-loop 跑完，收敛后把 review 报告（含 reviewer 汇总 must_fix + CONVERGED 结论）作为交付物落盘到 changes/ 下对应 review 文件（如 review-clarity.md，verdict: APPROVED）。这是 skill 改造新增的一步（见 §10）。
  2. agent 调 CW submit(clarify/detail) 时，该 review 文件已存在于 changes/ 目录。
  3. CW gate 跑 check_*.py，脚本自动发现 review 文件并通过校验。CW 不产桩、不理解 review 内容，只负责跑 check 脚本。
为何这不算造假：review 确实在 skill 阶段做了（reviewer subagent 跑过），skill 改造只是把原本 in-memory 的结果结构化落盘。
mid test commitHash 语义：test 阶段本身跑测试不产新 commit，mid test 提交的 commitHash 指向 dev 阶段产出的、被本次测试覆盖的 commit（即 testCase 所验证的代码版本）。GitValidator 校验该 commit 真实存在且属于本仓库，确保测试基于真实代码而非凭空声明。一个 testCase 可关联一个 commitHash（通常是对应 Wave 的 dev commit）。

§5.3 gate 执行规则

  • 多 checker（如 mid detail 的 4 个）：串行跑，全 pass 才 pass，任一 fail 则整体 fail（fail-fast，剩余不跑）。
  • gate 结果记录：每次 action 调用都追加 gateHistory（含 tier/result/report/ts），渐进式 action 标 progressive: true。
  • gateTier 透传：返回值的 gateTier 字段直接来自注册表，让 agent 知道本次 gate 强度。

§6 外部依赖分类与 Port 决策

依赖分类Port 决策
check_*.py 脚本Local-sub（自有可控，subprocess 调用）不做 Port。直接 subprocess 调用。稳定的内部契约。
git 命令True-external（第三方但极稳定）不做 Port。git CLI 是事实标准，签名固定。
_cw.json 存储In-process（文件 IO）不做 Port。直接 fs 读写。换 sqlite 是内部重构。
test-orchestrator 判定In-process（同进程函数调用）不做 Port。judgeByExpected 是纯函数，直接 import。
Pi SDK（registerTool 等）True-external（框架）不做 Port。SDK 升级时直接改 index.ts。
结论：CW 不引入任何 Port 抽象。零 Port = 零假设 seam（Seem 纪律）。代价（诚实标注）：没 checker seam，mid 被迫复用 full-* 的 check 脚本，两者 review 产物契约不同，需 CW 补 review 桩（§5.2）。若未来 checker 语义分化加剧，可能需 fork 脚本或加 CLI flag——YAGNI 的已知成本。
child_process 运行时约束：项目约束仅 pi-workflow（spawn）/ pi-subagents（execFileSync 只读 git）有 child_process 豁免。CW 是第三个用户：GateRunner 用 spawn 调 python check_*.py（与 pi-workflow 同模式）；GitValidator 用 execFileSync 调只读 git 命令（cat-file / merge-base / diff-tree，与 pi-subagents 同模式，无写副作用）。
lib/gates 去留：codebase 现有 lib/gates/（ReviewGate / TestFixLoopGate）是孤儿代码——src/index.ts re-export 但全 repo 无生产消费方。CW 引入新 gate 体系后，src/index.ts 移除 lib/gates 的 re-export（避免新旧 gate 体系并存混淆）。lib/gates 代码本身保留（不删历史代码），仅断开 re-export。

§8 核心模型：_cw.json 数据结构

CwTopic 是 CW 唯一持久化根（一个 topic 的完整状态，非 DDD aggregate）。

模型类型职责
CwTopic状态封装（持久化根）一个 topic 的完整状态，CW 唯一持久化根
CwStatus值对象（枚举）状态机节点
Wave实体dev 任务单元，含 committed commitHash
TestCase实体test 任务单元，含 status/actual/screenshot
GateHistoryEntry值对象gate 执行记录（不可变）
Evidence值对象收尾证据快照
GateTier值对象（枚举）gate 强度档位

CwTopic 结构

interface CwTopic {
  schemaVersion: number;            // 版本兼容，当前 1
  topicId: string;                  // cw- 开头加 slug
  slug: string;
  tier: "lite" | "mid";             // 锁定
  objective: string;
  workspacePath: string;
  createdAt: string;                // ISO
  status: CwStatus;
  planFormat?: string;              // "lite" | "mid-clarify" | "mid-detail"

  waves: Array<{
    id: string;                     // W1
    changes?: string[];             // technicalChanges id（lite）
    issues?: string[];              // issue id（mid）
    dependsOn: string[];
    parallelGroup?: string;
    committed: string | null;       // commitHash 或 null
  }>;
  testCases: Array<{
    id: string;                     // E1 / T1.1
    layer: string;                  // lite: mock|real；mid: unit|integration|e2e|perf-chaos
    scenario: string;
    steps: string;
    expected?: { url?: string; text?: string };  // lite 结构化（机器重算基准）
    assertion?: string;             // mid 自然语言（信声明，不重算）
    executor: string;
    status: "pending" | "passed" | "failed";
    actual?: object;
    screenshotPath?: string;
    commitHash?: string;
    judgedAt?: string;
  }>;

  gateHistory: Array<{              // 唯一真相源（含 pass/fail/渐进事件）
    phase: string; action: string; gate: string;
    tier: GateTier; result: "pass"|"fail"; ts: string;
    report?: string; progressive?: boolean;
  }>;

  evidence?: {
    topicId: string;
    finalStatus: CwStatus;
    coverage?: number;              // lite=机器重算实际通过率；mid=agent 声明通过率（语义不同）
    gateHistory: Array<{ phase: string; gate: string; result: string; ts: string }>;
    testSummary?: { total: number; passed: number };
  };
  coverage?: number;
}

type CwStatus = "created" | "planned" | "clarified" | "detailed"
  | "developed" | "tested" | "retrospected" | "closed";
type GateTier = "weak-structural" | "medium-git" | "medium-coverage" | "strong-recompute";
schema 演进：CwStore 加载时做版本兼容（schemaVersion，当前 1），缺失字段给默认值。statusHistory 已删除：原与 gateHistory（pass 子集）+ status（当前态）双重记账，信息完全可推导，单一真相源用 gateHistory。coverage tier 分化：lite = 机器重算实际通过率（strong-recompute，可信）；mid = agent 声明通过率（medium-coverage，未独立重算，语义弱于 lite），消费方需意识到差异。

§10 Skill 收口改造（D-007 落地，[REVISIT] 降级）

[REVISIT of D-007]：原 D-007 要求各 skill 删路由章节 + 改 description。reviewer（红队）指出 D-007 删路由与 D-011 改名用同一推迟理由但待遇相反，且 CW 硬强制在状态机（D-009），删路由是软约束。降级决定：MVP 只做入口 skill + description 映射句，不删路由。理由：CW 状态机已硬强制防跳过，skill 路由即使保留，agent 照走也会撞状态机。

§10.3 技术现实约束：Pi 的 skill 通过 available_skills 自动暴露，无"私有 skill"机制。收口靠 description 设计让 agent 不想跳过，而非 看不到。

§12 JSON schema 草案（D-006 落地，skill 产 CW 解析）

D-006 要求 plan/clarify/detail 阶段产出必须是结构化 JSON（CW 直接解析拆任务）。现有 skill 产 .md（给人读）+ .html（可视化），需新增 .json（给机器读）作为第三产出物。

方案 A（选定）：skill 在 final 步骤额外产 JSON（.md 给人 review + .json 给 CW 解析 + .html 给可视化）。
方案 B（否决）：CW 从 .md 解析——markdown 解析不可靠。
方案 C（否决）：agent 中介转换——agent 手工转易出错，且 agent 不应承担格式转换职责。

§12.2 plan.json（lite-plan 产出）

interface LitePlan {
  format: "lite";                  // 必须 === topic.tier，否则 CW 拒绝
  objective: string;
  waves: Array<{
    id: string;                    // W1
    changes: string[];             // technicalChanges id
    dependsOn: string[];
    parallelGroup?: string;
  }>;
  testCases: Array<{
    id: string;                    // E1
    layer: "mock" | "real";
    scenario: string;
    steps: string;
    expected: { url?: string; text?: string };  // 机器重算基准
    executor: string;
  }>;
}

§12.3 clarify.json（mid-plan 产出）

mid clarify 阶段产 requirements + architecture（设计文档），不含任务清单（任务在 detail）。主要是 tier 确认 + 交付物引用。

interface MidClarify {
  format: "mid-clarify";            // 必须 === "mid-clarify"
  objective: string;
  deliverables: {
    requirements: string;          // 文件路径，如 "requirements.md"
    systemArchitecture: string;    // "system-architecture.md"
  };
  // 不含 waves/testCases（mid 任务在 detail 阶段解析）
}

§12.4 detail.json（mid-detail-plan 产出）

interface MidDetail {
  format: "mid-detail";             // 必须 === "mid-detail"
  objective: string;
  waves: Array<{
    id: string;                     // W1
    issues: string[];               // issue id（mid 以 issue 为任务单元）
    dependsOn: string[];
    parallelGroup?: string;
  }>;
  testCases: Array<{
    id: string;                     // T1.1（mid 格式）
    layer: "unit" | "integration" | "e2e" | "perf-chaos";
    scenario: string;
    steps: string;
    assertion: string;              // 自然语言断言（信声明，不重算）
    executor: string;
  }>;
  deliverables: {
    issues: string;
    nonFunctional: string;
    codeArchitecture: string;
    executionPlan: string;
  };
}

§12.5 plan-parser 解析规则

  • CW submit(plan/clarify/detail) 入参含 JSON 文件路径或内联 JSON
  • plan-parser 按三种 schema 解析，校验 format 字段 === topic.tier 锁定值
  • pass 时 waves/testCases 写入 _cw.json（clarify 不写任务清单，只确认 tier）
  • schema 版本演进：JSON 顶层可加 schemaVersion 字段，plan-parser 向后兼容
跨系统依赖补并：JSON schema 草案是 D-006 可行性的关键前提。skill 不产 JSON 则 CW 解析链路断裂。skill 改造（产 JSON）与 review 落盘、description 映射同批，归 skill 收口改造（§10）。

§13 遗留物处理与衔接（内化带来的遗留代码）

§13.1 删除

coding-execute.js workflow 脚本

extensions/coding-workflow/workflows/coding-execute.js

旧的编码执行 workflow 脚本。设计分析确认其存在根本性问题：

  • coding-execute skill：派发指导
  • agent：实际 subagent 派发
§13.2 保留

check_execute.py

extensions/coding-workflow/skills/coding-execute/scripts/check_execute.py

coding-execute skill 自带的执行后机器门，读 test-results.json 验证。隶属 coding-execute skill，不属 test-orchestrator。

跨格式能力：同时支持 lite 用例 ID（E 加数字，如 E1）和 mid 用例 ID（T 加用例号.序号，如 T1.1），因此 coding-execute skill 在 lite 和 mid 路径都能复用同一个执行后机器门。

与 CW test gate 的关系（两者并存，分工不同）：

  • check_execute.py：coding-execute skill 内部用，执行后校验 test-results.json 结构完整性 + 用例覆盖（weak-structural，skill 自我检查）
  • CW test gate（内化的 judgeByExpected / 信声明）：CW 状态机用，决定 developed→tested 流转 + gatePassed

衔接链路：agent 调 coding-execute skill 执行 Wave → skill 跑 check_execute.py 自检 → agent 拿结果调 cw test 传 cases 数组 → CW test gate 判定。两者不冲突。需注意 test-results.json 与 cw test 的 cases 数组数据来源一致（同一批 test-runner 跑的结果）。
