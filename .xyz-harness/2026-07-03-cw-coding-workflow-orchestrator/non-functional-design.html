D-016 node:sqlite 三维度分析

存储层切换（文件 tmp+rename → node:sqlite + 关系表）是本次 NFR 最大变量。下表把 #1 在三个关键维度的前后变化单列，原子性已实测、并发待 V1 骨架验证。

数据完整性 · 风险下移

原子性下沉给 sqlite

事务边界由每个 action handler 的 BEGIN/COMMIT/ROLLBACK 承担；迁移走 PRAGMA user_version + ALTER TABLE（关联 #11），不再 JSON deserialize 补默认。已实测：崩溃事务不污染 db。原子性从「自实现 tmp+rename」下沉为 sqlite 天生，风险显著降低。

并发控制 · 新不确定性

DatabaseSync 同步 API 连接管理

同进程多 session 同时操作同一 _cw.db（迷雾 #14）——DatabaseSync 是同步 API，多连接的文件锁行为、WAL 可用性、busy_timeout 重试语义需在骨架验证。状态机 guard（#2）天然幂等；跨进程竞争靠 BUSY 重试吸收；同进程同步 API 多连接竞争需串行化（per-topic 单连接），否则同步阻塞调用栈可能死锁——V1 骨架验证哪条路径成立。

兼容性 · 三层版本边界

node:sqlite experimental

Node < 22.5 不可用；22.5-23.3 需 --experimental-sqlite flag；≥ 23.4 免 flag。除 engines.node 锁 ≥ 22.5 外，README 与加载时 catch 块对 flag 缺失给明确错误提示（区分「版本不够」与「flag 未开」）。CW 仅用 SQL 标准（CREATE/INSERT/BEGIN/SELECT），不依赖 experimental 语法糖，API 变更对用法影响可控。

详细副作用分析 · 逐 issue × 7 维度

写量规则：✅ 维度给一行理由；⚠️ 维度按「风险 / 影响 / 缓解 / 残余」四字段展开（#1 的数据/并发/兼容三维度详见上方 D-016 卡片，此处交叉引用避免重复）。

#1

CwStore node:sqlite DAO + 事务

方案 A · 手写 DAO
  • 安全 ⚠️风险：agent 输入的 slug/topicId/commitHash/JSON 进 SQL，字符串拼接则注入。影响：所有写库 handler。缓解：DatabaseSync.prepare(sql).bind(...args) 全量参数化 + DAO 层 lint 禁拼接模式。残余：0。
  • 数据 ⚠️事务边界 + ALTER 迁移 + 回滚 → 见 D-016 数据完整性卡片（原子性已实测）。关联 #11。
  • 性能 ✅CW 负载极低（单 agent 串行，每 topic 一 db 文件，单 action 几条 SQL），sqlite 事务开销远非瓶颈，无需索引/缓存优化。
  • 并发 ⚠️DatabaseSync 连接管理 + WAL + busy_timeout → 见 D-016 并发卡片 + V1 骨架验证。
  • 稳定性 ✅sqlite 崩溃恢复天生（事务原子性 + WAL），disk-full/IO 错误 sqlite 自处理并以 throw 上报，CW 转 infra-error 即可。
  • 兼容性 ⚠️node:sqlite experimental + 三层 Node 版本边界 → 见 D-016 兼容性卡片。
  • 可观测 ⚠️事务 COMMIT/ROLLBACK 落结构化日志（topicId + action + 耗时）；gate_history 表是天生审计表（sqlite3 CLI 可查），不需额外指标。
#2

状态机 guard 组织

方案 A · 声明式转换表
  • 安全/数据/性能/并发/稳定/兼容 ✅guard 是纯内存无状态只读函数：入参已由 typebox schema 在 tool 接口层校验；guard throw 在任何写之前；内存转换表查询；无外部依赖；内部规则。无副作用。
  • 可观测 ⚠️guard throw 须区分两类语义——illegal state transition（线性违例）vs previous phase incomplete（跨阶段级联未满足），错误码/消息分离让 agent 知道是「调错 action」还是「上阶段没跑完」。无告警（agent 自处理）。
#3

GitValidator 失败语义

方案 A · 逐条容错
  • 安全/数据/性能/并发/兼容 ✅execFileSync 无 shell 注入面（args 直传 argv，不经 shell）；只读 git 不直写 _cw.db（失败结果经 store 事务记录）；本地 git 快；只读无共享态；git CLI 签名稳定。
⚠️ 稳定性
  • 故障场景git 可执行文件缺失（ENOENT）/ 仓库损坏 / 非零退出码。须区分「commit 无效（业务 fail，该 task 记 fail 继续）」vs「git 基础设施不可用（infra-error，throw 中止 action）」。
  • 降级 / 重试不降级、不重试（git 本地调用确定性，重试无意义）——infra-error 直接 throw 让 agent 知道是环境问题。
  • SLA无影响。
⚠️ 可观测
  • 日志fail 的 task/case 须在返回的 nextAction 明确列出（failureReason 含具体哪项校验挂了：cat-file 不存在 / merge-base 不属仓库 / diff-tree 空 commit）。
  • 审计gate_history 记录 fail 项。
#4

gate 注册表编码

方案 A · 声明式数组
  • 全维度无风险 ✅内部声明式配置 + 通用执行器：纯内存只读注册表；gate_history 在 #1 事务内追加；fail-fast 是纯逻辑。无外部边界、无副作用、无可观测新增需求（gate_history 表已是审计载体）。
#5

plan-parser JSON 校验

方案 A · typebox Value.Assert
⚠️ 安全
  • 风险agent 可喂任意/超大/深嵌套 JSON 给 plan/clarify/detail action，恶意 100MB JSON 撑爆内存或深嵌套爆栈。
  • 影响范围plan/clarify/detail 三个 handler。
  • 缓解解析前 size guard（reject > 1MB，CW 交付物体积可控）；typebox Value.Assert 校验 schema；禁用递归深度异常的输入。
  • 残余0。
⚠️ 数据完整性
  • 事务/并发/迁移N/A（解析无写）。
  • 回滚策略format !== topic.tier 时 throw（D-003 tier 锁定），解析失败不写库，状态不变。
⚠️ 性能
  • 预期负载 / 关键路径单次解析体积小；超大 JSON 解析是唯一延迟点。
  • 优化size guard 提前拒绝（与安全同因同缓解）。
  • 并发/稳定/兼容/可观测 ✅无状态纯函数；typebox Value.Assert 是纯校验；内部 schema；校验错误消息明确（哪个字段缺/类型错）。注意：Value.Assert 在 Pi 运行时可用性需 V3 骨架验证。
#6

GateRunner subprocess 错误处理

方案 A · stdout + exitcode
⚠️ 安全
  • 风险topicId/topicDir 是 agent 输入，若含路径遍历（../）可能把 check 脚本指向 .xyz-harness/ 之外。
  • 影响范围所有调 check_*.py 的 gate（plan/clarify/detail/closeout）。
  • 缓解create-topic 时校验 slug 格式（^[a-z0-9-]+$）；topicDir 固定解析为 .xyz-harness/<slug>，reject 含 ../绝对路径。
  • 残余0。
⚠️ 性能
  • 负载 / 延迟每次 gate 一次 spawn；python 启动 ~100ms + 脚本执行，可接受。
  • 优化设 subprocess 超时（如 60s），超时 kill 标 infra-error。
⚠️ 稳定性
  • 故障场景(1) python 缺失 ENOENT；(2) 脚本 crash；(3) 超时；(4) verdict 行与 exitcode 矛盾。前三种 infra-error，第四种也是 infra-error（脚本契约破裂）。
  • 降级 / 重试不降级、不重试。
⚠️ 兼容性
  • API 变更依赖 check_*.py 的 stdout verdict 行格式（[check] machine check: N/N passed → PASS/FAIL）。脚本格式变则解析断。
  • 客户端影响 / 灰度check 脚本与 CW 同仓（extensions/coding-workflow/scripts/），版本天然耦合；解析器对 verdict 行的契约测试 pin 住格式。
⚠️ 可观测
  • 日志infra-error vs business-fail 在 gate_history 的 result/report 字段可区分（infra 标 infra-error，business 标 fail + report）。
  • 审计gate_history。
  • 数据 ✅subprocess 是只读检查（check_*.py 读文件产报告，不写 _cw.db），写库仍由 CW handler 在事务内做。
  • 并发 ✅每次 gate 独立 subprocess，无共享态。
#7

review 桩跨 skill 契约

方案 A · 预检 + hint
  • 安全/数据/性能/并发 ✅文件存在性 stat；不写；快；只读。
⚠️ 稳定性
  • 故障场景skill 改造（D-007-REVISIT）未完成或 review-fix-loop 未跑 → check_*.py 因缺 review 文件 fail，agent 困惑（不知是设计问题还是文件缺失）。
  • 降级方案CW 预检 review 文件缺失时返回结构化 hint（「review 文件缺失，请先跑 skill 阶段的 review-fix-loop」），非裸 check 报错。
⚠️ 兼容性
  • API 变更CW ↔ skill 跨组件契约——CW 不产 review 文件，依赖 skill 改造批次（D-007-REVISIT）产。两者必须同批发布。
  • 客户端影响旧 skill（未改造）+ 新 CW → mid clarify/detail gate 必 fail（hint 引导）。
  • 灰度 / 回滚CW 与 skill 收口改造同 PR/同 changeset 发布。
  • 可观测 ✅hint 本身就是可观测载体（明确告知缺什么文件、下一步做什么）。

#10

渐进入参数组统一

内部循环
  • 安全/性能/并发/稳定/兼容/可观测 ✅内部循环处理数组，无新外部边界；循环开销可忽略；无共享态；纯逻辑；内部；progress 字段透传。
  • 数据完整性 ⚠️事务边界：action 级事务包裹整个 task 循环；fail task 不 throw 只记 failureReason，故不触发 ROLLBACK，已成功 task 持久化（等价 per-task 语义，D-005 渐进式）。注：若 store 调用因 sqlite 错误抛异常则整批 ROLLBACK（含已成功 task），这是可接受的一—sqlite 错误属基础设施故障非业务 fail。与 #3 GitValidator 逐条容错语义一致。
#11

schemaVersion 演进

user_version 迁移
  • 安全/性能/并发 ✅迁移函数是 CW 内部 DDL，无外部输入；一次性 init 迁移开销可忽略；init 时单连接无并发。
⚠️ 数据完整性
  • 事务边界ALTER TABLE 迁移须在事务内，失败 ROLLBACK 保留原 schema 数据。
  • 迁移方案PRAGMA user_version 记录版本，CwStore 初始化按版本号顺序跑迁移函数链（user_version 0→1→2...）。
  • 回滚策略迁移失败 → 事务 ROLLBACK，db schema 不变；.xyz-harness/ 受 git 追踪，灾难回滚 = git checkout 旧 db（D-016 审计特性）。
⚠️ 稳定性
  • 故障场景迁移函数 bug 留下半迁移 db（user_version 改了但 ALTER 没全跑完）。
  • 降级 / 重试不降级；迁移在单事务内要么全成要么全败，无需重试。
⚠️ 兼容性
  • API 变更旧 _cw.db（user_version 0）被新 CW（期望 1）打开 → 自动向前迁移。无降级路径（新 db 用新 schema，旧 CW 打不开）。
  • 数据兼容向前兼容（旧→新自动迁移），不向后兼容（新→旧需 git checkout 旧 db）。用户跨 CW 版本切换时降级靠 git。
⚠️ 可观测
  • 日志迁移执行落日志（from version → to version + 耗时），让用户知道 db 被迁移过。
  • 审计gate_history 不涉及；迁移日志是独立观测面。

来源#1 并发 / 迷雾 #14

验证什么(1) 每次 action 打开 DatabaseSync 后关闭是否泄漏文件句柄；(2) 同进程两个 session 同时打开同一 _cw.db 是否死锁/阻塞；(3) PRAGMA journal_mode=WAL 与 PRAGMA busy_timeout=N 在 node:sqlite 是否可设置且生效。

预期结论per-action open/close 简单可靠（CW 负载极低，无需连接池）；WAL+busy_timeout 可用，意外并发由 BUSY 重试吸收；若无 WAL 支持，降级为单连接串行（CW 串行假设下可接受）。
