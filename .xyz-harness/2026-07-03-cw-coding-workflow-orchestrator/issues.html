P0 阻塞项（必须先做）

#1 CwStore 基于 node:sqlite 的 DAO + 事务封装
P 级: P0类型: 模块Blocked by: 无推荐强度: Strong
CW 持久化层（原架构 §8 _cw.json）。D-016 决定改用 Node 内置 node:sqlite + 关系表模式。CwStore 封装 sqlite 连接 + 事务 + DAO（topic/wave/test_case/gate_history 分表）。渐进式 dev/test action 多次更新，事务保证原子性（sqlite 天生，测试已验证：崩溃事务不污染）。关联 §8（数据模型，需修正为关系表）、§7（不变式：渐进式提交原子性 → 现归 sqlite 事务）。
为什么是 P0（阻塞）：_cw 存储是 CW 唯一持久化根。CwStore DAO 是所有 action handler 的依赖，不定下来上层无法开工。

方案对比

方案 A · 手写 DAO（每表一模块，事务在 service 层）— ✓ 选定
store.ts 导出 CwStore 类封装 DatabaseSync 连接；每张表一个 DAO 函数集；事务用 BEGIN/COMMIT/ROLLBACK 包裹多表写；handler 调 service 层组合 DAO + 事务。
优点：零额外依赖；事务边界清晰；SQL 可优化；符合项目「纯逻辑提取独立模块」测试规范。缺点：手写 SQL 样板；SQL 语法错误运行时才发现。
方案 B · 引入轻量查询构建器（kysely）
编译期类型检查 SQL，重构友好；但引第三方依赖违背 node:sqlite 零依赖初衷，kysely 对 node:sqlite 适配未必成熟。
方案 C · 扁平 SQL 字符串（handler 直写）
SQL 散落各 handler；事务边界乱；难测试；违反「状态层集中」。仅适用原型。
取舍：选 A。node:sqlite 选定即为零依赖（D-016），B 引 kysely 违背初衷。CW 表结构稳定 + 查询简单（按 topicId 整体读改写），手写 SQL 样板可控。DAO 分模块 + service 层事务，DAO 可注入 mock DatabaseSync 单测。

验收标准

  • AC-1.1 [正常]: CwStore 事务提交后数据持久（trace: UC-2 各 action）
  • AC-1.2 [异常]: 事务中抛错 ROLLBACK，数据回滚（崩溃不污染已验证）
  • AC-1.3 [边界]: 首次 create（空库）能建表 + 插入
  • AC-1.4 [兼容]: schema 演进用 ALTER TABLE（关联 #11）
  • AC-1.5 [隔离]: DAO 可注入 mock DatabaseSync 做单测（纯逻辑提取）
#2 状态机两重校验组织模式
P 级: P0类型: 架构Blocked by: 无推荐强度: Strong
§4.2 定义两重校验：① 线性 expectedStatus（currentState ∈ expectedStatuses[action]）；② 跨阶段 gatePassed 级联（test 需 dev 全 Wave committed；retrospect 需 test 全 case passed）。这两道校验如何组织——集中在 state-machine.ts 还是散在各 handler——决定 handler 骨架与防跳过的可靠性。关联 §4（状态机）、§7（不变式：状态机线性 + 跨阶段级联）。
为什么是 P0（阻塞）：状态机是 CW 的核心价值（D-009 主强制点）。组织模式定不下来，8 个 action handler 的骨架无法开工。

方案对比

方案 A · 声明式转换表 + 通用 guard 引擎（集中）— ✓ 选定
state-machine.ts 导出 TRANSITIONS: Record<Action, {expectedStatuses, nextStatus, requirePhaseComplete?}> 声明式表 + 通用 guard(action, topic)，各 handler 入口调 guard()。优点：防跳过逻辑集中单点；加 action 只改表；两重校验同函数顺序执行不漏；可测试（纯函数）。缺点：handler 与 guard 耦合靠约定，编译器不强制。
方案 B · 各 handler 自己实现校验（分散）
handler 自包含；但 8 个重复代码，第二道校验（级联）易在某 handler 漏写；改规则要改 8 处。违反 D-009 防跳过可靠性。
方案 C · 类装饰器 / 中间件模式
编译期强制；但 Pi handler 是普通函数，装饰器加魔法；TS 装饰器配置复杂；调试难。轻量工具不值得。
取舍：选 A。声明式转换表是状态机社区标准（XState/redux 思想），§4.2 已给完整表，声明式最自然。「约定耦合」缺点靠 SDK 契约测试覆盖（每个 handler 必须调 guard）。方案 B 的 8 处重复 + 易漏第二道校验是真实防跳过风险，不可接受。

验收标准

  • AC-2.1 [正常]: 合法转换（created→planned）guard 通过（trace: UC-1 AC-1.2）
  • AC-2.2 [异常]: 非法转换（created→tested）guard throw illegal state transition 不跑 gate
  • AC-2.3 [异常]: developed→test 但 dev 有 Wave 未 committed，guard throw previous phase incomplete
  • AC-2.4 [异常]: tested→retrospect 但 test 有 case 未 passed，guard throw previous phase incomplete
  • AC-2.5 [契约]: 每个 action handler 入口调 guard（SDK 契约测试覆盖）
  • AC-2.6 [语义]: 首次有效 dev/test 提交触发状态流转，后续渐进提交只更新 _cw.db（UC-3 AC-3.5 / UC-4 AC-4.5）

P1 核心项

#3 GitValidator 失败语义
P 级: P1类型: 模块Blocked by: #2推荐强度: Strong
GitValidator 用 execFileSync 调只读 git 命令（cat-file 存在 / merge-base 属仓库 / diff-tree 非空）校验 commitHash 真实性（§7）。某条校验失败时整个 action 该 throw 还是只把该 task/case 记 fail 继续其他？决定渐进式提交的容错语义。

方案对比

方案 A · 该 task/case 记 fail，继续其他（逐条容错）— ✓ 选定
GitValidator 返回结构化结果（每条 pass/fail + 原因），handler 把 fail 的 task/case 置 failed、记 gateHistory，action 不 throw，返回 nextAction 让 agent 看到哪些 fail 去修。优：符合渐进式语义，agent 拿明确反馈。缺：gate 判定要聚合多 task/case 结果。
方案 B · 整个 action throw（fail-fast）
实现简单；但一个坏 commit 导致整批失败，agent 无法区分；违背 D-005 渐进式提交意图。
取舍：选 A。符合 D-005 渐进式语义 + §7「该 task 记 fail」（已隐含逐条）。部分失败应继续，nextAction 反馈失败项让 agent 针对性重试。

验收标准

  • AC-3.1 [正常]: 合法 commit 校验 pass（trace: UC-3 dev/test）
  • AC-3.2 [异常]: 不存在 commitHash → 该 task fail，其他继续
  • AC-3.3 [异常]: commit 不属本仓库 → 该 case fail
  • AC-3.4 [反馈]: fail 的 task/case 在 nextAction 明确列出
  • AC-3.5 [约束]: mid test commitHash 必须能在 _cw.db wave 表追溯到已 committed 的 dev commit（UC-4 AC-4.3）
#4 gate 注册表编码模式
P 级: P1类型: 模块Blocked by: #2推荐强度: Strong
§5.2 给了 11 行 tier×phase→checker 映射表（如 lite×dev→GitValidator medium-git）。如何编码进 gates.ts 决定 gate 执行器的实现模式。多 checker 场景（mid detail 4 个 check）需串行 fail-fast（§5.3）。

方案对比

方案 A · 声明式数组 + 通用执行器 — ✓ 选定
GATE_REGISTRY: Array<{tier, phase, checkers: Checker[], tier: GateTier}>；通用 runGate(tier, phase, topic) 查表 → 串行跑 checkers → fail-fast → 记 gateHistory。优：数据与执行分离；加 gate 只改数组；fail-fast 单点实现；gateTier 透传自然。缺：Checker 需统一接口。
方案 B · 按 phase 注册函数链（命令式）
灵活；但注册散落，tier×phase 映射不直观（读多处才知道全貌），fail-fast 每个链自己实现。
取舍：选 A。声明式数组让 §5.2 的 11 行表 1:1 映射成代码，可读性最高。通用执行器单点实现 fail-fast + gateHistory + gateTier 透传，符合 §5.3。

验收标准

  • AC-4.1 [正常]: 单 checker gate pass/fail 正确（trace: UC-2 plan gate）
  • AC-4.2 [异常]: 多 checker（mid detail 4 个）串行，任一 fail 则 fail-fast
  • AC-4.3 [记录]: 每次调 runGate 追加 gateHistory（含 tier/result/ts）
  • AC-4.4 [透传]: 返回值 gateTier 字段来自注册表

P 级: P1类型: 模块Blocked by: 无推荐强度: Strong
§12 定义 3 套 JSON schema（LitePlan/MidClarify/MidDetail）。plan-parser.ts 需解析并校验入参 JSON。项目已用 typebox（Tool 参数 schema）。关联 §12、§7（tier 不可变：format===tier 校验）。

#6 GateRunner subprocess 错误处理
P 级: P1类型: 边界Blocked by: #4推荐强度: Worth exploring
GateRunner 用 spawn 调 python check_*.py（§6 第三用户）。subprocess 可能：非零退出（check 发现问题）/ 超时（脚本卡死）/ crash（python 不存在）。如何区分「check 判定 fail」（业务结果）vs「subprocess 异常」（基础设施问题）？

方案对比

方案 A · 解析 stdout verdict 行 + 非零退出码双信号 — ✓ 选定
现有 check 脚本输出 [check] machine check: N/N passed → PASS/FAIL。GateRunner 解析末尾 verdict 行作为 gate 结果，exit code 辅助；超时 kill + 标 infra-error。优：复用现有脚本输出契约；区分业务 fail vs infra error。缺：依赖脚本输出格式稳定性。
方案 B · 只看 exit code
最简单；但无法区分业务 fail vs infra crash，丢失 report。
取舍：选 A。现有 check 脚本已有稳定输出格式（mid-plan 已验证），解析 verdict 行拿到 report 让 agent 知道哪里 fail。exit code 作辅助校验（verdict 与 exitcode 矛盾时标 infra-error）。超时 kill 标 infra-error。

验收标准

  • AC-6.1 [正常]: check pass → gate pass（trace: UC-2 各 plan gate）
  • AC-6.2 [异常]: check fail（verdict FAIL + exit 1）→ gate fail + report 透传
  • AC-6.3 [异常]: subprocess crash（python 缺失）→ infra-error throw
  • AC-6.4 [异常]: 超时 → kill + infra-error
#7 review 桩跨 skill 契约保障
P 级: P1类型: 流程Blocked by: 无推荐强度: Strong
§5.2 review 桩机制：check_*.py 期望 changes/ 下对应阶段的 review 文件（verdict APPROVED），该文件由 skill 阶段（review-fix-loop）落盘，CW 不产。风险：skill 改造未完成或 review-fix-loop 未跑，则 check_*.py 因缺 review 文件 fail，agent 困惑。

方案对比

方案 A · CW 检测 review 文件缺失时给明确 hint + 信任 skill 改造 — ✓ 选定
CW gate 跑 check_*.py 前预检 review 文件是否存在，缺失则返回结构化错误「review 文件缺失，请先跑 skill 阶段的 review-fix-loop」。优：agent 拿可操作 hint；不造假 stub。缺：agent 仍需理解 review-fix-loop 流程。
方案 B · CW 自动生成 review stub（造假）
check 不报错；但违背 §5.2 决策，review 没真跑就 APPROVED 是造假，违背诚实执行原则。
取舍：选 A。CW 不造假。预检 + hint 让 agent 知道是流程缺失非设计问题。skill 改造（D-007-REVISIT）保证 review-fix-loop 跑过则文件在。

验收标准

  • AC-7.1 [异常]: review 文件缺失时 CW 返回明确 hint（非裸 check 报错）
  • AC-7.2 [正常]: review 文件存在且 APPROVED 时 check 正常跑

#15 skill 收口改造批次
P 级: P1类型: 流程Blocked by: 无推荐强度: Strong
§10 定义 skill 收口改造 4 子项：① 新增入口 skill coding-workflow（指导 agent 调 cw create-topic 后按 nextAction 走）；② 各阶段 skill description 顶部加「对应 CW action: xxx」映射句；③ review-fix-loop 收敛后落盘对应阶段 review 文件（verdict APPROVED），满足 check_*.py 前置；④ lite-plan/mid-plan/mid-detail-plan 的 final 步骤产 plan.json/clarify.json/detail.json（§12）。原覆盖表整体归 #7，但 #7 只覆盖 review 桩契约，本 issue 收口 4 子项整体。关联 G2、§10、§12、D-007-REVISIT、D-006。
为什么是 P1：G2 MVP 验收项明确要求「agent 工具箱中只有 coding-workflow tool + 新增入口 skill coding-workflow」。无 blocked_by（与 #1~#8 主体代码并行，skill 改造只依赖 CW action 接口契约）。

方案对比

方案 A · 按 tier 分批改造（先 lite 链路 3 skill，再 mid 链路 4 skill）— ✓ 选定
第一批改 coding-workflow（新增入口）+ lite-plan（description+JSON+review 落盘）+ coding-execute（description），跑通 lite 端到端；第二批改 mid-plan + mid-detail-plan + coding-retrospect + coding-closeout，复用第一批模板。优：与 CW 按 lite→mid 上线节奏对齐；lite 先跑通早验证 G2；每批可独立验证。缺：跨批过渡期 description 映射不一致，靠 §10.4 映射表人工对齐。
方案 B · 一次性批量改造（7 skill × 4 子项同批）
收口一次完成，description 立即一致；但约 28 处改动风险集中，与 CW 主体代码并行 skill 频繁 churn 易冲突，mid JSON 格式未验证就要改易返工。
方案 C · 按子项分批（先全 description，再全 JSON，再全 review）
每子项跨 skill 统一；但每 skill 被改 3 次 churn 大，子项间有依赖（JSON 依赖 description 锁定，review 依赖 JSON phase 识别）分批反而要协调。
取舍：选 A。CW 本身按 lite→mid 分档上线（G2 MVP 只验收 lite+mid），skill 改造与上线节奏对齐最自然。lite 端到端早验证 G2，mid 复用 lite 模板。跨批 description 不一致靠 §10.4 映射表过渡。B 约 28 处风险集中且 mid JSON 易返工；C 每 skill 改 3 次 churn 大且有依赖。

#16 coding-execute skill 适配 CW

P2 重要项

#9 nextAction 数据结构
P 级: P2类型: 模块Blocked by: 无推荐强度: Worth exploring
CW 每次 action 返回 nextAction 指导 agent 下一步。结构需含 skill（调哪个 skill）、guidance（文字提示）、可能的 expectedCases（test 阶段列出待测用例）。
推荐方案：扁平结构 {action?, skill?, guidance, waves?, testCases?} —— action/skill 可空（test 阶段无专用 skill 则 skill 空）；guidance 总有；waves/testCases 按阶段填充。理由：agent 消费简单，按字段取值判断下一步。编码时在 state-machine.ts 与转换表同处定义（§3 已定 nextAction 组装归 state-machine.ts）。
#10 渐进式提交入参数组统一
P 级: P2类型: 模块Blocked by: 无推荐强度: Strong
dev/test 入参是数组（D-005：长 1=单/N=批量）。handler 内部如何统一处理。
推荐方案：内部统一循环（长 1 是 N=1 特例，不分快路径）。理由：渐进式语义本就是逐个累计判定，单/批量逻辑本质相同（循环 + 累计），分支优化快路径增加复杂度无收益。handler 遍历数组 → 逐个跑 gate → 累计 gatePassed。
#11 schemaVersion 演进与 ALTER TABLE 迁移
P 级: P2类型: 模块Blocked by: 无推荐强度: Worth exploring
D-016 改用 sqlite 关系表后，schema 演进从「JSON deserialize 缺字段补默认」变为「sqlite ALTER TABLE」。未来 schema 升级（如加 full 路径状态）如何管理。
推荐方案：sqlite 的 user_version PRAGMA 记录 schema 版本。CwStore 初始化时检查 user_version，按版本号顺序跑迁移函数（ALTER TABLE / CREATE INDEX）。MVP 不引入复杂迁移框架（YAGNI），手写迁移函数链够用（user_version 0→1→2...）。未来真要升版本（如 D-010 full 接入触发）才加对应迁移函数。

延后理由：D-011 已决推迟到 CW 稳定后，避免改名与 CW 实现交织增加变更面。CW MVP 用现状名 + §10.4 映射表过渡。

#14 多 session 并发写同一 _cw.db ?
状态: ?（迷雾）类型: 边界
NFR C-1 关注多 session 隔离（禁模块级 let）。CW 无模块级可变状态（合规），但 _cw.db 是文件级共享。若两个 agent session 同时操作同一 topic（一个 dev 一个 test），D-016 的 sqlite 连接默认无并发锁。但 sqlite 本身有 BUSY 处理机制（WAL 模式 + 重试），比原文件方案 tmp+rename 的「A 读→B 读→A 写→B 写」逻辑覆盖问题更可控。
当前假设：单 agent 串行操作一个 topic（CW 实际运行模式，agent 不会并行操作同一 topic 的不同阶段）。验证方式：在 #1 实现后用集成测试模拟顺序调用确认无覆盖。若未来发现真并发需求（如 GUI 多窗口），再加乐观锁（version 字段 + CAS）。为何暂不展开：当前架构假设单 agent 串行，无证据显示需要并发。展开需先确认 GUI/多 agent 场景是否真存在，属未来需求（D-010 full 接入或 xyz-agent GUI 集成时再看）。
