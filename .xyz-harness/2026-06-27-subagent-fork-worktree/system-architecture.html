verdict: pass mode: refactor 4 新增模块 · 9 修改模块 无 Port（D-019 删 GitPort） crashed 新终态
TL;DR · 核心结论
  • 核心计算 = 技术流程编排（非业务规则编排）→ 维持现有三层架构（外壳/Runtime/Core）做增量增强。编排复杂度集中（SubagentService 编排 + session-runner 执行），无领域规则膨胀，三层铁律（Core 零 Pi 依赖/依赖单向）被充分复用。
  • 4 个新增模块：SessionContextResolver（Core，纯函数 D-014）/ WorktreeManager（Runtime，git 生命周期+collectPatch+scan+gitRun）/ FinalizedMarker（Runtime，.finalized sidecar）/ alive-store（Runtime，D-021 反哺 pid 探活载体）。
  • 状态机加 crashed 新终态：5 态（running/done/failed/cancelled/crashed）全不可逆；crashed 不经 tryTransition（重建推断态）；三分支检测（.cancelled/.finalized/都无）→ D-021 扩展四分支（pid 探活）。
  • 无 Port：D-019 删 GitPort（红队 deletion test 击穿，git CLI 极稳定 seam 价值近零），git 收口到 WorktreeManager 内部私有 gitRun helper。
  • 关键决策：D-014 SCR 纯函数化（forkFrom 上移 session-runner）/ D-017 finalizeRecord 三件套独立 try/catch + collectPatch 先行 / D-021~D-024 反哺（pid 探活/数据黑洞/类型裂缝/删活 worktree）。

设计立场

核心计算是技术流程编排（非业务规则编排）：把"委派子任务"这个意图，编排成"解析上下文 → 创建/隔离执行环境 → 执行 → 回收改动 → 标记状态"的确定性 pipeline。无复杂业务规则引擎。分层决策：维持现有三层架构（外壳/TUI/Runtime/Core）做增量增强——编排复杂度集中（SubagentService 编排 + session-runner 执行），无领域规则膨胀趋势；现有三层铁律（Core 零 Pi 依赖、依赖单向）被充分复用；新增模块按"是否有外部副作用 / 零依赖可单测"归入对应层。

统一语言（Ubiquitous Language）

术语定义

mainCwd主 agent 的 cwd。恒定，用于 subagent session 存储目录编码。

crashed新增终态：进程被 kill -9/OOM/断电打断，无 finalized/cancelled 标记的子 agent 状态。

关键决策

D-014 · REVISIT of D-009

SessionContextResolver 纯函数化

张力：归 Core 编排但调 forkFrom 需 pi SDK（打破 Core 零依赖）+ forkFrom 有 IO 副作用（打破纯函数）。

决策：SCR 只返回 fork 意图数据（真纯函数），forkFrom 调用留 session-runner。3 帧交叉验证击穿初稿矛盾。

D-017 · GAP-E6

finalizeRecord 三件套独立 try/catch

决策：顺序 ⓪collectPatch→①completeRecord→②archive→③finalized+cleanup，三件套各自独立 try/catch。

collectPatch 先于 completeRecord（patch 要进 result）；archive 在副作用写之前（防 record 卡 running）。

crashed 特化

不经 tryTransition

违反：状态变更不经 CAS 收口。

合理：crashed 是重建推断态（进程已死，内存 record 不存在），非运行期转换。tryTransition 保护 running record 收尾竞争，crashed 不参与。

GAP-E5 特化

cleanup 挂 finalizeRecord 内

违反：cleanup 不在 run() finally（session-runner Core 层）。

合理：放 session-runner finally 会跨层（Core 调 Runtime）。放 finalizeRecord 符合分层 + 三终态路径全覆盖。
