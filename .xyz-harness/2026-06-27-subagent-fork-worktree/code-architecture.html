TL;DR · 核心结论
  • 4 新模块 + 8 扩展：新建 session-context-resolver（Core 纯函数）/ worktree-manager（Runtime，含 collectPatch 合并自 PatchCollector）/ finalized-marker + alive-store（Runtime sidecar 家族）。扩展 types/session-runner/subagent-service/record-store/session-file-gc/subagent-tool/index/execution-record。
  • 核心接线：pi SDK 经 SdkLike 鸭子类型（D-016 声明 forkFrom 静态 + createBranchedSession 实例，D-018 两级降级）；git CLI 收口 WorktreeManager.gitRun（D-019 无 GitPort）；无 port、无循环依赖。
  • D-017 finalizeRecord 时序：⓪collectPatch→①completeRecord→②archive→③finalized+cleanup，D-022 collectPatch 失败跳过 cleanup 保 worktree（数据黑洞防护）。
  • test-matrix 双来源：来源 A（7 UC × 正常/边界/异常/状态/并发/e2e）+ 来源 B（④NFR 9 条 代码测试 缓解项→用例，强制 integration）。
  • 追踪收敛：5 帧 9 K-gap（无 F/D），2 条 CROSS-VALIDATED（重建帧独立证实覆盖帧），全修。check_code_arch 7/8（仅 review 待 Step6）。
架构决策（非 hero）：4 必问决策点（目录粒度/契约深度/依赖严格度/异常覆盖）均被上游 confirmed 决策消除歧义——D-009 模块归层 / D-019 删 GitPort / ②§11 grep 严格边界 / D-026 行为测试。grilling = agent 自决，无需 ask_user。Core 零 Pi 依赖（SCR 纯函数 D-014），Runtime 编排 Core，无循环。

Deep Module 设计决策

★ WorktreeManager（#4，核心深模块）

删 → git 生命周期逻辑散 3 处

Interface: create/cleanup/scan/collectPatch（小表面积）

Depth（deletion test）: 删则 clean校验/add/软链/setupHook/diff/remove/branch-D/reaper 孤儿判据散到 Svc+index+runner ≥3 处。承担大量复杂度 → 深模块。

Seam/Port: 无外部 seam（D-019 删 GitPort）。内部 seam=私有 gitRun。git 属 Local-substitutable，不要 port（单测 mock gitRun）。

★ alive-store（#13，叶子深模块）

删 → sidecar+探活散 3 处

Depth: 删则 .alive 读写+pid 探活散到 record-store+session-runner+session-file-gc 3 处。

Port: 不要（node:fs+process.kill 内置）。

★ SessionContextResolver（#3，纯函数）

零副作用零 Pi 依赖（D-014）

Depth: 抽出价值=可单测+零 Pi 依赖+未来 spawn 后端可复用（D-003）。

Port: 不要（纯计算 In-process）。

finalized-marker（#5，薄但必要）

对称 tombstone sidecar 家族

Depth: 薄但保持 .cancelled/.finalized/.alive 三件一致 + 可单测。
