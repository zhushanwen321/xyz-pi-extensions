关键副作用（需下游关注）

★ D-022 数据黑洞（#7）

collectPatch 失败 → cleanup 删 worktree

worktree subagent 核心交付物是 git 改动。collectPatch 失败时若仍 cleanup（D-017 三件套 best-effort 连带）= patch 没生成 + worktree rm -rf + branch -D = 改动彻底丢失不可恢复。

缓解：collectPatch 失败→cleanup 必须跳过（保 worktree+分支）+ record.result.patchFailed:true + 路径。数据安全 > 回收完整性。AC-7.4 故障注入测试。

回灌 #7 AC-7.4（代码测试）
★ D-024 删活 worktree（#4/#9）

reaper 跨实例误删实例 A 活 worktree

reaper 看不到实例 A 内存 running record，若孤儿判据=「无对应 running record」→ 误把 A 正在跑的活 worktree 当孤儿清 → 子 agent 工作目录 rm -rf。比 crashed 误判严重一个量级（删数据）。

缓解：孤儿判据=终态标记 且无活 .alive（readAliveMarker+isProcessAlive from #13）。绝不删活 .alive 的 worktree。AC-4.4/9.4 跨实例故障注入。

回灌 #4/#9 AC（代码测试）
★ pid 复用（#12）

24h 隐形僵尸 record

A 死后 pid 被 B 复用 → .alive 探活返回活 → record 投影 externalInstance 最坏 24h（直到软超时）。不可见崩溃。

缓解：startedAt+24h 软超时兜底（AC-12.3）+ externalInstance 年龄分布指标（让僵尸可观测）。D-021 用户拍板：概率正确非确定，pi 长生命周期下复用窗口极小。

残余风险（已接受）
★ D-017 时序窗口（#5）

①②后③前无 .finalized 误判 crashed

D-017 时序 ⓪collectPatch→①completeRecord→②archive→③writeFileSync。真窗口：①②后③前 → 无 .finalized 但磁盘已 done → 三分支恒判 crashed（不读 recon.status）。

缓解：窗口极短（archive 与 writeFileSync 间）+ 纯显示误差（record 磁盘完整）+ crashed 是安全降级。原"UC-7 recon 兜底"不成立（三分支不读 recon.status）。

残余风险（已接受）
