① 澄清需求

subagent fork 上下文 + worktree 隔离 · ① 澄清需求 · business requirements

verdict: pass 3 业务目标 · 7 用例 upstream: 无（首阶段） decisions: D-001~D-008
TL;DR · 核心结论
  • 3 个业务目标：G1 子 agent 继承主 agent 对话上下文（fork:true）/ G2 子 agent 在独立 git worktree 运行（文件系统隔离）/ G3 fork+worktree 可组合且可观测性完整。
  • 7 个业务用例覆盖：fork 继承上下文 / worktree 隔离运行 / 组合使用 / worktree 清理+patch 回传 / 崩溃孤儿清扫 / list 可见性 / 崩溃状态正确标记。
  • 核心路线：in-process createAgentSession + SessionManager.forkFrom（D-001）+ git worktree add（D-002）+ session 存储恒用主 cwd 编码（D-004）+ finalized sidecar 崩溃标记（D-006）。
  • 纯后端系统（pi 扩展内部能力，无人类直接 UI），状态展示复用现有 TUI widget。依赖 pi CLI 核心（客户-供应商）+ git（遵奉者，极稳定）。
  • 关键不做：不 spawn 子进程（OS-1）/ 不推远端 merge（OS-2）/ 不做 bash sandbox 真隔离（OS-3）/ 不做 fork 自动脱敏（OS-4）/ 嵌套 worktree 首版禁止（OS-6）。

业务用例（7 个）

UC-1: fork 继承主 agent 对话上下文

Actor: 主 Agent（LLM 调 subagent 工具） · 关联目标: G1
主流程: subagent tool(fork:true) → resolveSessionContext(fork:true) 取主 session → forkFrom → createAgentSession restore 历史 → prompt(task) 带主历史执行 → finalize
替代: fork 指定从某 entryId（position:at|before，受 D-007 深度限制）/ 主 session 未 flush → 降级 from-scratch + 警告
异常: 主 session 路径取不到→finalizeFailed / forkFrom 源损坏→抛错→finalizeFailed / fork 深度>10→拒绝（D-007）
  • AC-1.1 [正常] fork:true 时子 agent 首次 prompt SessionContext.messages 非空且含主 agent 最近历史
  • AC-1.2 [异常] 主 session 为空时降级 from-scratch 启动，不崩溃
  • AC-1.3 [边界] fork 嵌套第 11 层被拒绝并报明确错误

UC-2: 子 agent 在独立 git worktree 中运行

Actor: 主 Agent（LLM 调 subagent 工具） · 关联目标: G2
主流程: subagent tool(worktree:true) → WorktreeManager.create 校验 clean + git worktree add → worktreePath 作 effectiveCwd → node_modules 软链 + setupHook → createAgentSession({cwd:worktreePath}) → 完成后 diff→patch + cleanup
替代: working tree 脏→报错提示 commit/stash（D-待定：报错优先）/ 非 git 仓库→worktree 隔离降级不可用
异常: git worktree add 失败→回退主 cwd 或 fail hard / 软链失败→子 agent 自行 install / 并发→branch/path 带 recordId+timestamp 保唯一
  • AC-2.1 [正常] worktree:true 时子 agent bash 工具 cwd = worktreePath（非主 repo）
  • AC-2.2 [异常] working tree 脏时 worktree 创建被拦截并给可操作提示
  • AC-2.3 [边界] 并发 2 个 worktree subagent，branch/path 不冲突

UC-3: fork + worktree 组合使用

Actor: 主 Agent · 关联目标: G3
主流程: subagent tool(fork:true, worktree:true) → WorktreeManager.create → resolveSessionContext(fork:true, cwd:worktreeCwd) → forkFrom(主session, worktreeCwd, 主cwd编码 sessionDir) → createAgentSession 既继承历史又在 worktree 跑
异常: UC-1+UC-2 异常并集；创建顺序：先 worktree（失败不 fork）→ 再 fork session（失败清 worktree）
  • AC-3.1 [正常] fork:true+worktree:true 时子 agent 既有主历史又 cwd=worktreePath
  • AC-3.2 [边界] session 落 subagents/<encoded-主cwd>/sessions/，RecordStore.collectRecords 能扫到（不落 worktree 编码目录）

UC-4: worktree 清理与 patch 回传

Actor: subagents 扩展（内部，subagent 完成后触发） · 关联目标: G2.1
主流程: finalizeRecord → diffWorktrees(git add -A → git diff --cached → patch 文本) → patch 路径塞 record.result → cleanup(worktree remove --force + branch -D，配对) → 主 agent 自行 git apply
异常: patch 捕获失败→写空 patch + 警告（改动可能丢失）/ cleanup 失败→静默吞 + 记日志，孤儿留 reaper / 无改动→空 patch 正常清理
  • AC-4.1 [正常] subagent 有改动时生成非空 patch 且 worktree+branch 已删
  • AC-4.2 [异常] subagent 无改动时 patch 为空但清理正常执行
  • AC-4.3 [边界] cleanup 失败不阻断 finalize（finalize 先于或独立于 cleanup）

UC-5: 崩溃/孤儿 worktree 清扫（启动期 reaper）

Actor: subagents 扩展（session_start 触发） · 关联目标: G2.2
主流程: session_start → WorktreeReaper.scan → git worktree list 找 pi-sub-* → 对照存活 records 判孤儿 → 清扫(remove --force + branch -D) → git worktree prune
异常: 清扫失败→记日志下次启动再扫（不阻断 session_start）/ worktree 被另一活 pi 占用→pid/时间戳判断跳过活态
  • AC-5.1 [正常] 进程被 kill -9 后重启，残留 pi-sub-* worktree 被清扫
  • AC-5.2 [边界] 正在运行的 worktree（关联活态 record）不被误清

UC-6: /subagents list 可见性

Actor: 主 Agent（调 subagent tool action:list） · 关联目标: G3
主流程: listHandler → RecordStore.collectRecords(limit, statusFilter) → 合并内存 running + 磁盘 session.jsonl 重建（reconstructAll） → 返回 SubagentRecord[]（含 fork/worktree 子 agent）
异常: worktree subagent session 在主 cwd 命名空间（D-004）→ collectRecords 能扫到，无异常
  • AC-6.1 [正常] worktree subagent 完成后 /list 能看到它（status=done）
  • AC-6.2 [边界] fork+worktree 组合的 subagent，session 落主 cwd 目录，list 可见

UC-7: 崩溃状态正确标记

Actor: subagents 扩展（重建 record 时） · 关联目标: G3
主流程: collectRecords → reconstructFromFile 检查 sidecar → 有 .cancelled→cancelled / 有 .finalized→recon 推导 done/failed / 都无→crashed（新增 ExecutionStatus）
异常: finalize 写 .finalized 后但进程写下一 entry 前死 → 降级用 recon stopReason
  • AC-7.1 [正常] 进程被 kill -9 后，崩溃的 subagent 在 /list 显示 crashed
  • AC-7.2 [边界] 正常完成的 subagent 写了 .finalized，重启后仍显示 done

约束（Constraints）& 不做（Out of Scope）

技术约束（仅记录，架构阶段展开）

C1-C7 硬约束

  • C1 in-process 模式（ADR-022），不 spawn 子进程（D-001, D-003）
  • C2 createAgentSession({cwd, sessionManager}) 接受任意 cwd + manager，自动 restore 历史
  • C3 SessionManager.create 的 cwd 与 sessionDir 是独立参数（D-004 技术基础）
  • C4 getSubagentSessionDir 返回子树，pi 原生 resume 物理扫不到（三层证据）
  • C5 kill-9/OOM/断电不可拦截，崩溃标记只能启动期检测（D-006）
  • C6 fork 全量复制有体积/安全风险，需深度限制 + 体积控制（D-007）
  • C7 worktree 隔离是文件系统级非进程内存级，bash 可逃逸（D-008）
既有 ADR 约束

遵守 / 需修订

  • ADR-022（in-process 执行模型）：遵守
  • ADR-024（L1+L2 持久化，L3 不做跨进程恢复）：遵守
  • ADR-001 决策 2（task=全部输入）：需修订——引入 fork 后契约不再绝对
编号不做项理由 / 决策
OS-1spawn pi 子进程双后端（L3 跨进程恢复）D-003 不做
OS-2worktree 保留分支推远端 + 主 agent mergeD-005 不做（接口预留 keepBranch）

OS-4fork redaction（自动脱敏密钥）D-007 文档化，不做自动脱敏

OS-6嵌套 worktree（worktree-of-worktree）首版禁止，仅 fork 可嵌套（受 D-007 限制）
