#1 types.ts 基础类型扩展 P0 · 模型
Blocked by: 无 推荐强度: Strong R2 反哺: G1 声明位置 / D-023 B7 修复

问题描述

为 fork/worktree 扩展跨层类型契约（四项互不依赖，同属 types.ts 高内聚）：ExecutionStatus 枚举加 crashed；SdkLike.SessionManager 加 forkFrom（静态，session-manager.ts:1434）声明 + createBranchedSession（实例方法，session-manager.ts:1286，返回 string|undefined + 原地 mutate）声明；SessionRunnerContext 加 mainCwd/mainSessionFile，cwd 语义收窄为 effectiveCwd（不加 gitPort）。

R2 反哺（B7 修复 D-023）：SubagentRecord 投影类型加 externalInstance?: boolean 字段——跨实例 running-elsewhere 用此投影标志，不污染 ExecutionStatus 联合类型（保持 {running, done, failed, cancelled, crashed}，不加 __external）。

R2 G1 关键修正：forkFrom 命中 SdkLike.SessionManager 静态块；createBranchedSession 命中 AgentSessionLike.sessionManager 实例端鸭子类型块（与 appendCustomEntry 同范式）——避免加错接口。

为什么是 P0

不加 crashed 到 ExecutionStatus 联合类型，#2（STATUS_PRIORITY）和 #7（finalizeRecord）的类型引用直接编译失败。SdkLike 扩展是 #6 调 forkFrom/createBranchedSession 的类型前置。这是编译基石。

方案对比

方案 A: 增量扩展（沿用鸭子类型约定）— 推荐 / 选定
  • 模型: ExecutionStatus 追加 "crashed"；forkFrom 声明到 SdkLike.SessionManager 静态块，createBranchedSession 声明到 AgentSessionLike.sessionManager 实例端鸭子类型块；SessionRunnerContext 加字段。
  • 不新建文件，全部在现有 types.ts 内。

优点: 与现有鸭子类型范式一致（appendCustomEntry 先例），下游无需适配新导入路径；crashed 让 tryTransition target 守卫天然类型安全。

方案 B: 抽独立 session-context-types.ts — 放弃

缺点: 违反 YAGNI——WorktreeHandle 归 Runtime、fork 意图归 Core，强抽单文件制造跨层耦合；与"types.ts 统一跨层类型"约定相悖。

取舍: 方案 A。沿用统一 types.ts 约定（长期架构一致性），forkFrom/createBranchedSession 是声明既有 SDK 能力的 subset 非新抽象。方案 B 过早按层分包。

#2 状态机基础 — execution-record + record-store + session-reconstructor P0 · 模型
Blocked by: #1 推荐强度: Strong #12 扩展点: 三分支→四分支

问题描述

实现 crashed 新终态的状态机收敛口，跨三高内聚模块：execution-record tryTransition target 加 crashed + 新增 markReconstructedStatus(record, status) 收口方法；record-store STATUS_PRIORITY 加 crashed=1（同次编辑补 key——#1 加 crashed 后 Record<ExecutionStatus> 要求全 key），reconstructAll 改三分支检测（.cancelled→cancelled / .finalized→done|failed / 都无→crashed），全部经 markReconstructedStatus；session-reconstructor crashed 判定上移 record-store。

注：三分支「都无→crashed」由 #12（D-021）扩展为四分支（pid 探活插入），#2 先实现基础三分支。

方案对比

方案 A: markReconstructedStatus 收口 + 三分支在 record-store — 推荐 / 选定
  • execution-record 加 markReconstructedStatus(record, status)（跳过 CAS——重建态无内存 running record 可竞争）。
  • record-store.reconstructAll 三分支 if/else，每分支调 markReconstructedStatus。

优点: crashed 判定单点收敛（record-store），消除散落判定；markReconstructedStatus 让重建路径与运行期转换走不同收口，状态机不变式保持。

方案 B: session-reconstructor 保留 crashed 判定（现状下沉）— 放弃

缺点: 违反 D-013#1（crashed 上移）；三分支需调用方组合 tombstone+finalized 双盘读，record-store 读不到完整态。

取舍: 方案 A。crashed 三分支需 tombstone + finalized 双盘读，两者都在 record-store 扫描域内；判定上移让单点拥有完整信息（长期可维护性）。方案 B 留下判定散落长期债。
