TL;DR

  • Goal 是 7 状态生命周期状态机（active/paused/blocked + 4 终态），显式转换表 VALID_TRANSITIONS 保障合法性
  • 3 层架构：engine（纯函数，零 Pi 依赖）→ adapters（Pi 桥接，按事件拆 6 handler）→ service+projection（协调+渲染）
  • task 合并到 todo extension（跨扩展读取），goal_manager 10 action → goal_control 2 action
  • budget 单一检查点在 persistState（对齐 Codex SQL CASE），删除 maxTurns/stall 自动终态

2. 设计立场

核心计算：Goal 是一个带 budget 约束的生命周期状态机。核心 = 状态转换 + budget 计量 + context prompt 注入。

分层决策：3 层（engine / adapters / service+projection）。engine 是纯领域层（零 Pi 依赖），adapters 是接口层，service+projection 是应用层。不套 DDD 四层——核心计算是纯领域规则，不是复杂业务流程编排。

10. 挑战与决策

D-A1: event-adapter 按事件类型拆分（6 handler + 薄路由）。理由：变化轴不同。
D-A2: service.ts 保持单文件 ~300 行。理由：删 10 action 后够用。
D-A3: 跨扩展 API 保持 duck-typed。理由：可选特性，undefined=降级。
D-A4: 显式转换表 VALID_TRANSITIONS。理由：与权限三分层双重保障。
D-A5: BudgetConfig 只留 tokenBudget + timeBudgetMinutes。理由：对齐 Codex。

特化决策

违反什么为什么合理触发变化怎么办
engine 通过 adapter 间接依赖 todoengine 零 Pi 依赖不变；ProgressInput 是注入 DTO只改 adapter
VALID_TRANSITIONS 与权限三分层重复双重保障两处都更新
persistState 同时做 persist + budget单一检查点（对齐 Codex）只改 persistState
