TL;DR

  • 用户可靠交付目标：任意时刻可暂停/恢复，暂停不耗预算；agent 卡住可报告阻塞等待介入。
  • 终态必然：删除"按轮数/停滞自动终止"，资源预算单一检查点兜底不漏触发；完成须自证。
  • 单一任务源：Goal 不再内嵌任务，统一 Todo 管理（含验证任务）；复杂目标可先规划后执行。
  • 职责三分层：agent（完成/阻塞）、用户（暂停/恢复/清除）、系统（资源耗尽），对齐 Codex 心智模型。

1. 业务目标（Business Goals）

背景：Pi 的 Goal 扩展提供「持久化自主循环」。相对 Codex 有架构优势，但存在三个影响「用户能否可靠地把目标交给 agent」的业务级差距：
  1. 用户无法干净地叫停：无"暂停"状态，中途插入指令会让目标状态混乱。
  2. 资源耗尽可能漏触发：预算检查分散在事件处理器，存在空转烧资源风险。
  3. 两套任务系统打架：Goal 内嵌任务与 Todo 并存，需提示词硬规则禁用一个，心智负担重。

目标树

  • G1 用户可可靠交付目标并随时干预
    • G1.1 干净暂停与恢复，不丢上下文 — 量化: 续跑 ≤1 turn 停止/恢复；任务清单 0 丢失
    • G1.2 agent 卡住有明确信号 — 量化: 阻塞可见延迟 ≤1 turn；阻塞态预算消耗 0
  • G2 目标必有明确终态，不无限空转
    • G2.1 资源约束唯一兜底，单一检查点不漏 — 量化: 耗尽后 ≤1 检查点转终态；漏触发=0
    • G2.2 完成须先完成全部任务（含验证）并给证据 — 量化: 违规完成拦截率=100%
  • G3 任务管理单一来源
    • G3.1 Goal 与 Todo 协作不冲突 — 量化: Goal 内嵌任务残留=0
    • G3.2 复杂目标可先规划后执行 — 量化: plan→goal 衔接 ≤1 步用户动作
  • G4 职责边界清晰可预期 — 量化: 越界操作拦截率=100%

达成路线

目标路线/策略对应用例
G1.1用户主导"暂停"状态，暂停/恢复对称且不丢上下文UC-2

G2.2完成审计：完成前完成全部任务（含验证）并附证据UC-4
G3.1合并两套任务系统为单一 Todo；Goal 只读快照UC-1, UC-4

2. 业务用例（Use Cases）

用例图见上方 Hero。每个 UC 含前置/主流程/替代/异常/后置 + AC。

UC-1: 复杂任务规划执行（plan → goal → todo → audit）

Actor: 开发者（主）、Agent（协助） · 关联目标: G3.1, G3.2, G2.2
主流程: 发起 goal → Agent 判复杂建议规划 → 规划完成自动衔接 → 建任务（含验证）→ 持续推进 → 全完成声明完成附证据
替代: 简单任务 Agent 跳过规划直接执行 · 异常: 遇阻转 UC-5 · 后置: 完成终态附证据
  • AC-1.1 [正常] 复杂目标可走"规划→执行→完成"全流程，最终完成态附证据
  • AC-1.2 [异常] 规划/执行中遇阻可转 UC-5，不卡死
  • AC-1.3 [边界] 简单目标可不规划直接执行，衔接不被强制

UC-2: 用户叫停与恢复目标（暂停/恢复）

Actor: 开发者 · 关联目标: G1.1
主流程: pause → 暂停态停续跑 → 插入指令/改任务 → resume → 活跃继续
替代: 从 blocked 恢复（重检资源） · 异常: 崩溃后暂停态保持；resume 时已超预算 → 拒绝恢复
  • AC-2.1 [正常] 暂停期间 agent 不续跑、不消耗预算
  • AC-2.2 [正常] 恢复后基于最新任务清单继续，不丢上下文
  • AC-2.3 [边界] 崩溃重启后暂停态保持，不自动恢复
  • AC-2.4 [异常] 恢复时若已超预算，拒绝恢复（不转活跃）

UC-3: 资源耗尽自动终止

Actor: 预算系统（自动） · 关联目标: G2.1
主流程: 持续消耗 → 统一检查点检测耗尽 → 转资源耗尽终态（区分 token/时间）→ 通知原因
替代: 70%/90% 预警引导收尾 · 异常: agent 忘完成 → 不自动替它完成，预算兜底
  • AC-3.1 [正常] 预算耗尽必然终止，不漏触发（单一检查点）
  • AC-3.2 [正常] 终态区分 token 耗尽与时间耗尽
  • AC-3.3 [边界] agent 忘记完成时不被自动"帮"完成

UC-4: Agent 自主完成并自证

Actor: Agent · 关联目标: G2.2, G3.1
主流程: 确认全任务（含验证）完成 → 声明完成附证据 → 校验任务完整性 → 完成终态
替代: 关联规划则软提醒核对步骤（非硬阻断） · 异常: 未全完成/未建/验证跳过 → 拒绝
  • AC-4.1 [正常] 全部任务（含验证）完成后可声明完成并附证据
  • AC-4.2 [异常] 任务清单为空/未加载/验证未完成时拒绝完成
  • AC-4.3 [边界] 验证任务不可取消，常规任务可取消视作完成
  • AC-4.4 [边界] Todo 未加载时降级运行，完成检查拒绝（完成受限，非不可用）
  • AC-4.5 [边界] plan 步骤未全执行但任务全完成仍允许完成（plan 软提醒，任务清单是唯一硬检查）

UC-5: Agent 报告阻塞并等待用户介入

Actor: Agent（报告）、用户（介入） · 关联目标: G1.2, G4
主流程: 报告阻塞附原因 → 阻塞态（与暂停对称）→ 任务仍可改 → 用户介入后恢复
替代: 用户判断无价值转 UC-6 · 异常: 非活跃态拒绝；崩溃后阻塞态保持
  • AC-5.1 [正常] 阻塞态不续跑、不消耗预算，等用户介入
  • AC-5.2 [正常] 用户可恢复阻塞态目标
  • AC-5.3 [边界] 非活跃态下报告阻塞被拒绝
  • AC-5.4 [边界] 崩溃重启后阻塞态保持，不自动恢复

UC-6: 用户主动清除目标

Actor: 开发者 · 关联目标: G4
主流程: 清除指令 → 取消终态（不可逆）→ 不续跑
替代: 有未终态目标时发起新目标被拒 · 异常: 已终态再清除幂等
  • AC-6.1 [正常] 清除后进入取消终态，不再续跑
  • AC-6.2 [边界] 存在未终态目标时发起新目标被拒

5. UI/UX 场景（Interface Scenarios）

注：交互界面为编辑器内状态提示 + 命令面板 + /goal 指令，无独立页面级 UI。

交互流程

  • 发起：/goal <目标>（可带预算）→ 活跃态，状态栏显示目标+剩余预算
  • 暂停/恢复：/goal pause → 暂停态、停续跑；/goal resume → 活跃
  • 查看状态：状态栏始终反映状态 + 剩余预算
  • 预警：70%/90% 主动提示
  • 终止通知：资源耗尽/完成时通知原因（含证据）
  • Agent 触发态变化（UC-4/UC-5）：完成展示证据摘要，阻塞高亮提示阻塞原因
  • 错误/拒绝反馈：操作被拒（非活跃态、resume 超预算、完成检查失败等）有明确回显+建议

状态可见性

  • 每状态（活跃/暂停/阻塞/完成/资源耗尽/取消）有对应显示
  • 暂停 vs 阻塞外观可区分（触发主体不同）
  • 资源耗尽终态区分 token / 时间耗尽

7. 约束（Constraints）

业务约束

  • 对齐 Codex 心智模型：不依赖"轮数/停滞自动终止"，信任 agent 自主完成/报告阻塞 + 资源预算兜底
  • 职责三分层不可越界（agent 不能替用户清除目标，用户指令优先）

技术约束（仅记录不展开）

  • TypeScript + Pi 扩展 API；会话持久化（无独立数据库）
  • 跨扩展协作用鸭子类型编程式接口（不合并扩展）
  • 事件串行、工具执行可能并发的并发模型

8. 不做（Out of Scope）

  • 不做自动完成：agent 忘声明完成时，系统不替它完成（预算兜底终止）
  • 不做按轮数/停滞自动终止：删除 maxTurns/maxStall 终态路径
  • 不保留两套任务系统：Goal 内嵌任务废弃，不与 Todo 并存
  • 不做 plan 完成审计硬校验：plan 步骤核对是软提醒（唯一硬校验是任务清单）
  • 不在暂停期间冻结任务清单：暂停是 agent 停跑，任务仍可调整
  • 不做 /goal abort：语义与清除重叠，统一用清除
