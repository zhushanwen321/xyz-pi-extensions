Phase-Gate：3 层次严格防伪造

层次检查内容🔴 核心🟡 非核心
1. 脚本检查JSON 格式 + 最终 version 全部 passed/skipped✅✅
2. 一致性检查test_cases_template.json 与 Phase 2 commit 版本比对逐字节比对抽查关键字段
2. Case 与 Spec/Plan每个 case 是否和 spec.md、plan.md 定义一致逐条比对抽查
3. 跳过理由质疑每个 skipped case：为什么跳过？环境日志？必须提供环境日志文字说明即可
3. 测试结果真实性evidence 是否包含实际命令输出/断言结果必须有执行日志通过即可
3. 修复记录完整性fixed case 是否有 fix_description + bug_cause完整记录fix_description 即可
FAIL → 打回主 agent 修复 → 修复后直接重新提交 phase-gate（不回到 workflow loop）
如果 test_cases_template.json 被篡改（与 Phase 2 commit 不一致），需要回到 Phase 2 重新确认。

为什么 Phase 4 不需要 Review-Gate？

其他阶段（1/2/3）的 Review-Gate 是"审查产出物质量"。但 Phase 4 的测试-修复循环本身就是质量保障——每个 case 必须通过或充分跳过。
Phase-Gate 的严格防伪造检查替代了 Review-Gate 的角色，且更深入（一致性比对、跳过理由质疑、修复记录审查）。

E2E 测试最佳实践（嵌入测试节点指导）

实践说明
