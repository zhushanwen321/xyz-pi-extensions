# E2E 优化落地可行性分析

## 最终排序

| 优先级 | 优化项 | 可行性 | 改动范围 |
|:------:|--------|:------:|---------|
| P0 | 用例分级回退 | 完全可行 | 2 个 md 文件 |
| P0 | 失败智能分析 | 可行（限制额外上下文） | 1 个 agent.md |
| P1 | Flaky 诊断 | 可行（仅诊断不重试） | 1 个 agent.md |
| P1 | 组前健康检查 | 可行 | 1 个 agent.md |
| P2 | L2>L3 信任规则 | 可行 | 1 个 agent.md |
| 暂缓 | 基线版本管理 | 缺确认环节 | — |
| 不落地 | 独立集成测试 Stage | 破坏依赖+联动成本高 | 已替代为组前健康检查 |
| 不落地 | DB 组间重置 | 与依赖模型根本冲突 | — |

---

## P0：用例分级回退 — 完全可落地

### 方案

当前回退规则是"任何 L1/L2/L4 失败 → 全部回退"。改为按严重程度分级：
- P0（阻塞级）失败 → 回退编码
- P1（重要级）失败 → 不回退，记录 FAIL，Stage 14 reviewer 评估
- P2（一般级）失败 → 记录 WARN，不阻塞

### 改动范围

- `xyz-harness-e2e-test-plan/SKILL.md` 的回退规则章节
- `harness-e2e-tester/agent.md` 的回退判定规则章节
- 返回格式新增 `done_with_failures` status

### 兼容性

- Gate 12 检查 `PASS|FAIL|SKIP` 关键词 — 不冲突（报告仍然包含这些关键词）
- Stage 14 reviewer 需要补充提示：P1 FAIL 不等于可忽略，仍需评估风险

### 风险

P1 失败可能带 bug 进入生产。应对：Stage 14 reviewer 必须审视所有 FAIL。

---

## P0：失败智能分析 — 可落地，需限制上下文

### 方案

用例失败时，只利用已获取的信息（curl 响应、DOM 快照、截图）做结构化根因分析，不做额外 CDP 调用。

### 关键约束

- 不增加额外网络请求
- 修复建议最多 2 句话
- 连续 3 个用例失败 → 停止执行（大概率是系统性 bug）

### 为什么不能做额外 CDP 调用

当前 agent 有严格的增量���入策略："每完成一个用例组，写入结果后丢弃原始数据"。失败分析的额外调用会消耗上下文。如果只有 1-2 个失败可以承受，5+ 个失败可能撑爆上下文。

---

## P1：Flaky 诊断（仅诊断，不重试）

### 方案

失败时判断是 FAIL(BUG) 还是 FAIL(ENV)：
- FAIL(BUG)：API 500、DOM 元素确实缺失 → 代码问题
- FAIL(ENV)：CDP 断连、API 超时 → 环境问题

### 为什么不做自动重试

有副作用的测试不能重试：
- "创建一个订单" → POST /orders → 数据库多了一条记录
- 重试 POST → 两条记录 → 后续断言基于"只有一条" → 必然失败
- 只有读操作（GET、页面导航）可以安全重试

判断重试是否安全本身就很复杂，不如只做诊断。

### 价值

主 agent 看到 FAIL(ENV) 时可以决定手动重跑 E2E，而不是回退到编码阶段。

---

## P1：组前健康检查 — 替代独立集成测试阶段

### 方案

不新增 Stage。在 harness-e2e-tester 每个测试组开始前，先做快速健康检查：

```bash
curl -s http://localhost:{port}/health || { echo "后端不可达，跳过 EG1"; continue; }
```

API 级问题在第一个测试组就能发现。

### 为什么不新增 Stage

**依赖关系破坏**：当前测试组围绕完整用户场景设计。把 L1+L4 抽出来先跑、L2+L3 后跑，会破坏组内上下文连续性。TC-1-02 的 L2 需要 TC-1-01 创建的数据和页面状态。

**Stage 编号联动**：16 个 Stage 编号被 stages.ts、harness-state.sh、所有 gate 脚本、所有文档引用。新增 Stage 改动面太大。

---

## P2：Layer 2 > Layer 3 信任规则

### 方案

当 Layer 2 和 Layer 3 结论矛盾时，以 Layer 2 为准。

原因：Layer 2（DOM/A11y 验证）是确定性的，Layer 3（VLM 视觉对比）准确率仅 ~45%。

规则：
- Layer 2 PASS + Layer 3 FAIL → 以 Layer 2 为准，Layer 3 标记 WARN
- Layer 2 FAIL + Layer 3 FAIL → 确认 FAIL

---

## 暂缓：基线版本管理

完整基线管理需要 baselines/ 目录、版本化命名、过期标记、更新审批。

核心问题：谁来判断"这是有意设计变更"？AI 不可靠（45%），当前流程中没有审批节点。等有真实使用场景后再设计。

---

## 不落地：独立集成测试 Stage

两个硬冲突：
1. 依赖关系破坏 — 拆分层级会破坏组内上下文连续性
2. Stage 编号联动 — 新增 Stage 需改 stages.ts、harness-state.sh、所有 gate、所有文档

已用"组前健康检查"替代。

---

## 不落地：DB 组间重置

与 e2e-test-plan 的核心设计根本冲突。测试组通过依赖关系图共享数据（EG1 创建用户，EG2 用该用户测试登录）。组间重置 DB 会摧毁依赖链。

数据隔离应在用例设计时考虑（不同用例操作不同实体），不是通过 DB 重置实现。
