# xyz-harness E2E 实现与业界差距分析

## 当前 E2E 流程完整链路

```
Phase 1:
  Stage 6: 编写 e2e-test-plan.md（主 agent + subagent 并行生成用例）
  Stage 7: 独立评审（harness-e2e-test-plan-reviewer，7 个维度）
  Gate 07: 验证评审报告无 MUST FIX

Phase 2:
  Stage 13: 执行 E2E 测试（harness-e2e-tester subagent）
  Gate 12: 四层验证（文件存在→内容完整→反伪造→L2 LLM 验证）
  Stage 14: 测试评审（harness-reviewer 评审测试结果）
```

## 已经做对的（5 项）

| 维度 | 业界标准 | xyz-harness 实现 | 评价 |
|------|---------|-----------------|------|
| 四层验证策略 | API + DOM + Visual + DB | Layer 1-4 完整实现 | 业界先进 |
| A11y Tree 优先 | role + name 优先 | 已实现，且禁止 Tailwind selector | 最佳实践 |
| Chrome 实例隔离 | 独立浏览器上下文 | --user-data-dir + 独立端口 | 正确 |
| 增量写入策略 | 防止上下文溢出 | 每组写入后丢弃原始数据 | AI 场景必须 |
| 伪造检测 | CI 不需要，AI 需要 | Gate 12 四层反伪造 | 独特且正确 |

## 需要改进的（5 项）

### 差距 1：缺少两层 E2E 策略

- 业界：Smoke（PR 级）+ 回归（预发布级）
- 当前：只有一层全量 E2E
- 影响：所有用例一视同仁，P2 视觉问题可能阻塞整个流程
- 建议：引入 P0/P1/P2 分级回退

### 差距 2：集成测试层缺失

- 业界：单元→集成→E2E，集成是 ROI 最高的层
- 当前：单元测试（Stage 12）→ 直接跳 E2E（Stage 13）
- 影响：E2E 承担了太多本应在集成层验证的东西
- 建议：组前健康检查（替代独立 Stage）

### 差距 3：缺少 Flaky 管理

- 业界：自动 retry + 隔离机制（Playwright 默认支持 retries）
- 当前：一次失败即 FAIL，无重试
- 影响：环境抖动可能导致假失败，触发不必要的回退
- 建议：flaky 诊断（仅诊断不重试），区分 FAIL(BUG) vs FAIL(ENV)

### 差距 4：缺少失败智能分析

- 业界：AI 自动聚合日志+截图+网络请求→输出根因+修复建议（70-80% 时间节省）
- 当前：只有 PASS/FAIL/SKIP 表格，失败分析由 Stage 14 reviewer 做
- 影响：主 agent 收到 FAIL 后不知道具体原因，只能回退编码
- 建议：失败时利用已获取信息做结构化根因分析

### 差距 5：Layer 3 无降级策略

- 业界：AI 视觉+像素对比组合，VLM 不准确时回退确定性验证
- 当前：只依赖 VLM，无兜底
- 影响：VLM 45% 精确检测率可能导致漏检或误报
- 建议：Layer 2 > Layer 3 信任规则

## 关键架构差异

### AI agent vs CI 执行模式

| 维度 | CI 管道 | AI agent（xyz-harness） |
|------|---------|----------------------|
| 执行方式 | 确定性代码 | LLM 推理 |
| 并行能力 | sharding，10 分钟跑完数百测试 | 严格串行 |
| 失败检测 | 断言匹配 | AI 语义判断 |
| 伪造风险 | 无 | 高（需要 Gate 12 反伪造） |
| 上下文限制 | 无 | 严格限制（增量写入） |

核心认知：不能简单套用 CI 的最佳实践。AI 的优势在于深层分析能力（视觉对比、智能分析），而非执行速度。应放大 AI 的理解力优势。

### 测试奖杯对 xyz-harness 的意义

当前 xyz-harness 从"单元测试"直接跳到"E2E 测试"，缺少中间的"集成测试"层。在测试奖杯模型中，集成测试是 ROI 最高的层（最多投资）。补齐集成测试层是最重要的结构性改进。

但由于 AI agent 执行模式的特点（串行、上下文有限），不适合新增独立 Stage。改为在 E2E 内部优化——组前健康检查 + 分级回退 + 智能分析，在不改变流程结构的前提下获得类似效果。
