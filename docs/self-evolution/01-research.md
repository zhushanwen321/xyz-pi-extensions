# 01 — 文章理论分析

> 对用户提供的 Agent 自我进化文献综述的逐项分析，映射到 pi 的技术架构。

---

## 1. 文章核心框架梳理

文章将 Agent 自进化研究按以下维度组织：

### 1.1 形式化理论层

**GVU 算子（Chojecki, 2025）**

```
新Agent = Updater( Agent, Verifier( Generator(Agent) ) )
```

三个角色的职责：
- **Generator**：Agent 执行任务，产出行为轨迹
- **Verifier**：对行为质量打分（规则、LLM-as-Judge、环境反馈）
- **Updater**：根据评判结果修改 Agent（提示词、技能、权重）

关键约束——**方差不等式**：只有当 Generator 和 Verifier 中的噪声足够小时，自改进才能持续正向。否则会出现"越改越差"的退化现象。

**对 pi 的启示**：Verifier 的质量是系统成败的决定因子。没有可靠的评判信号，自动化优化就是噪声放大器。pi 的 Verifier 必须基于 LLM-as-Judge，并结合人类审批作为最终门控。

### 1.2 五层进化架构

| 层级 | 进化对象 | 成本 | pi 可行性 |
|---|---|---|---|
| L1: 系统提示词优化 | CLAUDE.md / agent.md | 低（分钟级） | 高。CLAUDE.md 是文本文件，可被分析-建议-修改 |
| L2: 技能库扩充 | Skill 目录的 SKILL.md | 低-中（小时级） | 高。Skill 是 Markdown 文件，增删改成本低 |
| L3: 代码与测试进化 | Extension 源代码 | 中（过夜运行） | 中。Extension 是 TypeScript，可自动生成但需测试验证 |
| L4: 知识库增强（RAG） | 外部文档索引 | 中 | 低。pi 目前无 RAG 层 |
| L5: 模型微调 | 模型权重 | 高（天级） | 极低。pi 是客户端工具，不管理模型训练 |

**对 pi 的启示**：聚焦 L1 + L2。这两层成本最低、可控性最强、失败回滚代价最小。L3 作为中期目标，L4/L5 暂时不纳入规划。

## 2. 系统提示词自进化（L1）理论分析

### 2.1 DSPy：提示优化编译为搜索问题

**核心机制**：将 LLM 调用抽象为参数化模块，通过编译器自动搜索最优提示词和示例组合。工作流程：

```
定义程序签名 → 初始化候选提示 → 并行评估 → 选择最优 → （可选）蒸馏到小模型
```

**pi 适用性分析**：

DSPy 的"参数化模块"抽象不直接适用于 pi。pi 的系统提示词不是模块化的——它是一个整体的 CLAUDE.md + agent.md + skill SKILL.md 拼接。但 DSPy 的核心思想——"自动搜索最优提示"——可以映射为：

- **候选提示生成**：LLM Judge 生成 CLAUDE.md 的多个变体
- **并行评估**：在不同 session 或 subagent 中 A/B 测试变体
- **选择最优**：基于 session 成功率、token 效率等指标排名

**落地方式**：不是直接集成 DSPy 库，而是借鉴其思想，在 Phase 4 的 evolution-engine 中实现"提示词变体生成 + 评估"的简化版。

### 2.2 ZERA：从零初始化到结构化提示

**核心机制**：
- 八条通用评分标准（清晰度、完整性、任务适配性等）
- 结构化批评（不只是好/坏，而是具体改进建议）
- 快速收敛（少量示例 + 短迭代周期）

**pi 适用性分析**：

ZERA 的"冷启动"场景恰好对应 pi 的 CLAUDE.md 优化问题——当前的 CLAUDE.md 是人工编写的，缺乏系统性评估。ZERA 的结构化批评机制可以直接用于：

- 评估现有 CLAUDE.md 各章节的质量
- 生成改进建议（如"安全规则章节缺少对 bash 操作的约束"）
- 检查规则冲突（如两个规则给出矛盾的建议）

### 2.3 E-SPL：提示词与模型权重的协同进化

E-SPL 的核心价值在于**声明性知识**（提示词）和**程序性知识**（模型权重）的自然分工。但对于 pi 来说，模型权重不在控制范围内，因此：

- **直接可用的**：TrueSkill 评分系统用于进化选择（贝叶斯技能评级，比简单排名更稳定）
- **不可用的**：RL 权重更新、多提示词并行 rollout（pi 没有此基础设施）
- **可借鉴的**：TrueSkill 可用于 skill 的"技能评级"，决定哪些 skill 最有效

### 2.4 GEPA：遗传-帕累托提示优化

GEPA 的核心是**反思驱动的优化**——从试错中学习高级规则。这与 pi 的 session 日志分析天然契合：

- 从 session 的错误模式中提取"试错"信息
- 用 LLM 反思"为什么这些操作失败了"
- 生成高级规则（"处理 Vue 文件之前先检查组件库名称"）

## 3. 技能/SKILL 自进化（L2）理论分析

### 3.1 SkillX：三层技能层次

| 层级 | pi 对应 | 说明 |
|---|---|---|
| 战略计划层 | coding-workflow / harness-retrospect | 高层流程控制 |
| 功能技能层 | code-review-worktree, batch-tracer 等 | 可组合的中层功能 |
| 原子技能层 | whitespace-fixer, web-fetch 等 | 单一操作 |

pi 的 60 个 skill 已经自然形成了这种层次，但缺乏显式标注和组织。SkillX 的"从轨迹中提炼技能"可以直接用于 pi：分析 session 中重复出现的操作序列，提炼为新 skill 或优化现有 skill。

### 3.2 Ratchet：技能生命周期管理

这是文章中对 pi 最有直接指导意义的理论。四项卫生机制与 pi 的映射：

| Ratchet 机制 | pi 现状 | 差距 |
|---|---|---|
| 结果驱动的淘汰 | usage-tracker 有计数，无淘汰逻辑 | 需要增加"使用率 + 成功率"综合评分，低于阈值建议淘汰 |
| 技能数量上限 | 60 个，无上限 | 建议设定 80-100 的软上限，触发淘汰检查 |
| 元技能编写指导 | skill-creator 有指导，但偏创建不偏维护 | 需要增加"skill 质量标准"和"skill 退场标准" |
| 模式标准化 | 各 skill 格式不统一 | 可被元技能指导替代（Ratchet 的消融实验结论） |

**最关键的发现**：Ratchet 实验证明 LLM 自生成技能贡献为 +0.0pp，人类筛选后为 +16.2pp。这意味着自动进化**不能是全自动的**——LLM 只能做建议，人类必须做审批。

### 3.3 NSI：神经-符号技能归纳

NSI 的核心是让技能包含显式控制流（if-then-else、while、动态变量）。pi 的 skill 目前是纯 Markdown 文本（"触发条件 + 执行步骤"），不包含可编程逻辑。

NSI 对 pi 的启示：可以考虑在 skill 的 SKILL.md 中增加一个 **decision tree** 章节，用结构化的方式描述"何时做、为什么做"。这不是把 markdown 变成代码，而是在文本中显式地写出决策逻辑。

### 3.4 SkillForge：企业级闭环进化

三阶段管道直接映射到 pi 的进化框架：

```
SkillForge: 批量故障分析 → 技能诊断 → 技能优化 → 重新部署
pi 映射:   session 错误分析 → 定位到 skill/CLAUDE.md → 生成改进建议 → 人审后应用
```

关键词是 **"闭环"**——从发现到应用到验证再到发现，形成完整循环。

## 4. 与 pi 现有能力的对照矩阵

| 文章理论 | pi 当前对应能力 | 成熟度 | 差距 |
|---|---|---|---|
| Generator（行为生成） | session JSONL 完整记录 | 天然存在 | 需要结构化提取 |
| Verifier（质量评判） | 无 | 完全缺失 | 需要 LLM Judge + 人类审批 |
| Updater（自动修改） | 无（人工修改 CLAUDE.md/skill） | 完全缺失 | 需要工具支持（建议生成+diff预览） |
| 提示优化 L1 | CLAUDE.md 人工维护 | 有人工操作 | 缺自动化分析和建议 |
| 技能进化 L2 | skill-memory-keeper 被动记录 | 有记录无分析 | 缺统计分析+生命周期管理 |
| 使用统计 | usage-tracker（skill/agent 计数） | 基础可用 | 需要增加成功率、token 效率等维度 |
| 经验积累 | bug-fix-recorder, skill-memory-keeper | 有框架 | 需要自动化触发和综合 |

## 5. 理论收敛：pi 进化系统的设计原则

基于以上分析，提炼以下设计原则：

### P1: Verifier 优先原则

在 Updater 之前必须先建立可靠的 Verifier。没有 Verifier 的自进化 = 随机游走。pi 的 Verifier 应基于：
- **LLM Judge**（主评判）：用强模型（GLM-5.1）对行为质量打分和生成改进建议
- **统计信号**（辅助评判）：token 效率、重试次数、工具调用失败率等硬指标
- **人类审批**（最终门控）：所有自动修改在应用前必须经过人类确认

### P2: 先 L1 后 L2 原则

CLAUDE.md（系统提示词）的优化比 Skill 库的优化风险更低、效果更直接：
- CLAUDE.md 是单一文件，改动影响范围明确
- Skill 库涉及 60 个文件，改动可能产生交互效应
- 先通过 CLAUDE.md 优化验证 Verifier 的质量，再扩展到 Skill 库

### P3: 半自动优于全自动

Ratchet 的 +0.0pp vs +16.2pp 数据是铁证。自动生成的建议必须经过人类审批。但"半自动"不等于"人工做一切"——LLM Judge 的价值在于**结构化分析和建议生成**，人类的角色是**审批和优先级排序**。

### P4: 渐进式采集原则

信号采集应该在现有 extension 上渐进增强，而不是从头构建独立系统：
- 利用 usage-tracker 的事件监听模式
- 利用 skill-memory-keeper 的记录-总结-归档流程
- 利用 session JSONL 的现成格式
