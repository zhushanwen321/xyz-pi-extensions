# ADR 006: LLM Judge 作为独立子进程运行

**上下文**：Evolution Engine 需要调用 LLM 分析 session 信号数据并生成进化建议。有两个可选方式：(A) 在主对话中注入信号数据，让当前 AI 直接分析；(B) 启动独立 Pi 子进程（spawn-based），使用固定模型分析。

**决策**：选择方案 B——LLM Judge 运行在独立 Pi 子进程中，固定使用 `router-openai/glm-5.1` 模型。

**原因**：
1. 质量一致性：独立子进程可以硬编码高质量模型（glm-5.1），不依赖用户当前使用的模型。如果用户在低端模型上运行 `/evolve`，方案 A 产生的建议质量会显著下降。
2. 上下文隔离：信号数据可能达到 10-20KB，注入主对话会占用大量上下文预算。独立子进程有独立的上下文窗口。
3. 可验证性：子进程输出是严格 JSON，可以被验证和解析。方案 A 的输出嵌在主对话文本中，解析不稳定。
4. 与 subagent extension 的模式一致：团队已经理解了 spawn-based LLM 调用的机制。
