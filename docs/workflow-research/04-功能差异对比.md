# 04. 功能差异对比

> 目标:在前两节拆解基础上,做横向对比。**重点是"能力差异"** 和 **"为什么有这些差异"** —— 不只是列差异,而是揭示底层设计哲学的不同。
>
> 对比维度来自用户要求:
> 1. **用例**:典型场景的重合与差异
> 2. **核心领域设计**:领域概念、领域边界、领域服务的差异
> 3. **整体架构**:运行时模型、消息流、隔离模型的差异
> 4. **重要领域模型**:数据模型、状态机、持久化模型的差异

## 二、按用户要求维度的详细对比

### 2.1 用例差异

#### 2.1.1 共同用例(双方都擅长)

| 用例 | Claude Code | pi-workflow |
|------|------------|-------------|
| **批量审查**(每文件一个 agent) | ✅ | ✅ |
| **多角度交叉验证** | ✅(Adversarial / Judge Panel) | ✅(parallel + 多个 review agent) |
| **流水线**(review → fix → verify) | ✅ | ✅ |
| **AI 即时生成 workflow** | ✅(关键词触发) | ✅(`/workflow-generate` 显式) |
| **死代码清理**(循环模式) | ✅(Accumulate) | ✅(while + agent) |
| **Plan 草案多角度** | ✅ | ✅(parallel 多个 agent) |

#### 2.1.2 Claude Code 擅长的用例

| 用例 | 原因 |
|------|------|
| **超大规模(数百 agent / run)** | 默认 16 并发,1000 agent/run 硬限制;CC 公开材料里有 Bun 案例的数百 agent 实证 |
| **闭源生态集成** | 与 GitHub Actions、Claude API、Bedrock、Vertex AI、Foundry 集成 |
| **公开标杆案例** | Bun 重写,Klarna / CyberAgent 客户引用 |

#### 2.1.3 pi-workflow 擅长的用例

| 用例 | 原因 |
|------|------|
| **跨会话的长时间任务** | callCache + 7 态状态机 + JSONL rehydrate,run 可暂停数天再恢复 |
| **需要严格 token / cost 控制** | 三重预算 + 90% 警告 + 100% 终止 |
| **需要节点级重试/跳过** | retryNode / skipNode 精确控制 |
| **需要 AI 显式辅助生成 + 用户确认** | `/workflow-generate` + tmp dir + `/workflow save` 三步式 |
| **Pi 平台集成** | 与 goal / todo / coding-workflow / subagent 同进程,无跨进程 RPC |
| **GUI 渲染细节** | `_render` 描述符,适配 xyz-agent Vue 组件 |

#### 2.1.4 用例差异的本质

**Claude Code 把"规模"作为首要目标**(Bun 案例),**pi-workflow 把"可恢复 + 严格控制"作为首要目标**。
- CC 的 16/1000 限制是**为了防 runaway**,但允许"一次性冲完"
- pi-workflow 的 budget + 节点级控制是**为了可中断**,允许"边跑边控制"

### 2.2 核心领域设计差异

#### 2.2.1 领域概念差异

| 概念 | Claude Code | pi-workflow | 差异 |
|------|------------|-------------|------|
| **Workflow** | 脚本 + meta | 脚本 + meta | 同 |
| **Agent call** | `agent(opts)` 字符串 + options | `agent(opts)` 支持 3 种签名 | pi 更宽容 |
| **Parallel** | `parallel([...])` 支持数组 + 函数 | `parallel([...])` **只支持数组** | CC 灵活 |
| **Pipeline** | `pipeline([...stages])` | `pipeline([...stages])` | 同 |
| **Phase** | `phase(name)` 命名阶段 | `phase(name)` 命名阶段 | 同 |
| **Log** | 无显式 API | `log(msg)` → `{ type: "log" }` 消息 | **pi 独有** |
| **$ARGS** | 来自运行参数 | 来自 `/workflow run --args` | 同 |
| **$WORKSPACE** | 无 | `process.cwd()` | **pi 独有** |
| **$BUDGET** | 无 | `{ usedTokens, usedCost, maxTokens?, maxTimeMs? }` | **pi 独有** |
| **Schema** | JSON Schema | JSON Schema(简单 JSON.parse) | 同 |
| **CallCache** | 推断存在(从"resumable in same session"反推) | **显式持久化** | pi 显式 |
| **Trace** | 推断存在(从 `/workflows` 面板反推) | **显式** + 双重存储(内存 + JSONL) | pi 显式 |
| **Budget** | 16/1000 硬限制 + 推断有 token/cost | **三重预算 + 90% 警告** | pi 粒度更细 |
| **Run record** | 内存(同会话) | JSONL(跨会话) | **决定性差异** |
| **节点级控制** | 推断有 retry(未公开) | **显式 retryNode + skipNode** | pi 显式 |
| **Approval** | 第一次运行 / 按权限模式 | 无 | CC 独有 |
| **TUI 面板** | `/workflows` 进度 | `setWidget` + `/workflows` 面板 + 详情 | 相似 |
| **GUI 渲染** | 推测有 | **`_render` 描述符** | pi 显式 |

#### 2.2.2 领域边界差异

```
                Claude Code
   ┌──────────────────────────────────┐
   │ 主会话(Claude 持有)              │
   │   ↓ (触发)                        │
   │ Workflow Runtime(独立环境)        │
   │   ↓ (调度)                        │
   │ Subagent(独立上下文)              │
   └──────────────────────────────────┘

                pi-workflow
   ┌──────────────────────────────────┐
   │ Pi 主进程                         │
   │   ↓ (parentPort)                  │
   │ Worker 线程(Node worker_threads) │
   │   ↓ (spawn "pi")                 │
   │ Pi 子进程(--mode json -p)        │
   └──────────────────────────────────┘
```

**关键差异**:

| 边界 | Claude Code | pi-workflow |
|------|------------|-------------|
| **Runtime 位置** | "独立环境"(闭源,推测子进程或线程) | **Node Worker 线程**(同 Pi 进程) |
| **Subagent 位置** | 闭源内部 subagent(独立上下文) | **Pi 子进程**(`child_process.spawn`) |
| **消息协议** | 闭源 RPC | **显式 `parentPort.postMessage`**(Worker ↔ Main) + **JSONL stdout**(Main ↔ Pi 子进程) |
| **持久化** | 闭源 | **`pi.appendEntry` → Session JSONL** |

**本质差异**:pi-workflow 是**白盒**,每个边界都可观测、可调试、可扩展;Claude Code 是**黑盒**,运行机制靠推断。

#### 2.2.3 领域服务差异

| 服务 | Claude Code | pi-workflow |
|------|------------|-------------|
| **ScriptParser** | 闭源 | `extractMetaViaWorker` 在临时 Worker 中 import(沙箱化) |
| **AgentDispatcher** | 闭源 | `AgentPool` 显式 FIFO 调度 |
| **ResultCache** | 推断 | `callCache: Map<number, AgentResult>` |
| **SchemaValidator** | 推断用 zod 或类似 | 直接 `JSON.parse`,无校验 |
| **BudgetEnforcer** | 推断 16/1000 限制 | `checkBudget` + `scheduleTimeBudgetCheck` 显式 |
| **ApprovalGate** | 有(按权限模式) | 无 |
| **ProgressPanel** | `/workflows` | `setWidget` + 详情 overlay |
| **WorkflowStore** | `.claude/workflows/` | `config-loader.scanDirectory` 扫描 `.pi/workflows/` + `~/.pi/agent/workflows/` + `.tmp/` |
| **Reconstructor** | 不支持跨会话 | `reconstructState` 显式 rehydrate |
| **Notifier** | 推断 | `pi.sendMessage` 显式通知 |
| **StateMachine** | 推断(无公开) | 7 态精确机,`VALID_TRANSITIONS` 显式 |

### 2.3 整体架构差异

#### 2.3.1 隔离模型差异

| 隔离 | Claude Code | pi-workflow |
|------|------------|-------------|
| **Workflow ↔ 主会话** | 独立 runtime(推测) | **Node Worker 线程** |
| **Agent ↔ Workflow** | 内部 subagent | **Pi 子进程**(`child_process.spawn`) |
| **Agent ↔ Agent** | 内部调度 | **FIFO 队列**(`AgentPool.drain`) |
| **脚本 ↔ FS/Shell** | **不直接**(脚本只协调 agent) | **不直接**(脚本只协调 agent) |
| **Meta 提取 ↔ 主进程** | 闭源 | **临时 Worker**(10s 超时) |

**关键洞察**:pi-workflow 的隔离是**三段**(Pi → Worker → Pi 子进程),CC 是**两段**(推测 Pi → Runtime → Subagent)。三段意味着每层可独立失败、监控、扩展;两段更紧凑但更难调。

#### 2.3.2 消息流差异

**Claude Code(推断)**:
```
[Script] → [Internal RPC] → [Runtime] → [Subagent]
                                ↑
                          (闭源实现)
```

**pi-workflow(显式)**:
```
[User Script in Worker]
    ↓ { type: "agent-call", callId, opts }
[parentPort]
    ↓ (主线程 Orchestrator)
[AgentPool]
    ↓ spawn("pi", --mode json -p --no-session)
[Pi 子进程 stdout]
    ↓ 解析 JSONL
[AgentResult]
    ↓ postMessage { type: "agent-result", callId, result, cached }
[Worker]
    ↓ resolve(promise)
[User Script 继续执行]
```

**pi-workflow 的消息协议是"教科书级别"**:
- 明确的消息类型枚举
- 显式的 `cached: boolean` 区分 cache hit
- `_runId` 通过 `workerData.args._runId` 注入,确保错误也能回传
- Stale state 守卫(`orchestrator.ts:344-351`)防 race

#### 2.3.3 持久化模型差异

| 维度 | Claude Code | pi-workflow |
|------|------------|-------------|
| **持久化路径** | `.claude/workflows/` 项目级 + `~/.claude/workflows/` 全局 | `.pi/workflows/`(saved)+ `.pi/workflows/.tmp/`(tmp)+ `~/.pi/agent/workflows/` 全局 |
| **同名优先级** | 推断项目级优先 | **显式:tmp > project > user** |
| **Run 历史** | **不持久化**(内存态,exit 即丢) | **持久化**到 Session JSONL(`workflow-state` + `workflow-trace`) |
| **GC 策略** | N/A | **无 GC**(append-only,代码注释承认 JSONL 膨胀) |
| **跨 session 恢复** | **不支持** | **支持**(同 session 内 paused ↔ running 任意切换,跨 session 重新进入可 paused 状态恢复) |
| **重放机制** | N/A | callCache 按 callId 重放 |

**决定性差异**:CC 把"运行状态"视为瞬态,pi-workflow 把"运行状态"视为可恢复的持久态。这是**两种根本不同的设计哲学**:

- **CC 哲学**:workflow 是"一次性任务",完成即过;若失败,重新写脚本
- **pi-workflow 哲学**:workflow 是"长生命周期任务",可以暂停数天,可以跨 session 接管

#### 2.3.4 失败处理差异

| 失败 | Claude Code | pi-workflow |
|------|------------|-------------|
| **Agent 调用错误** | 推断 retry | **显式 3 次 retry + 指数退避 1s/2s/4s** |
| **Worker 脚本错误** | 推断 mark failed | **显式 3 次 retry + 指数退避,失败 mark `failed`** |
| **Budget 超限** | 推断终止 | **显式:`budget_limited` 终态**(token/cost)+ **`time_limited` 终态**(time) |
| **状态机非法转移** | 推断 throw | **显式 throw + 工具返回 isError** |
| **Workflow 不存在** | 推断抛错 | **显式:`api.sendUserMessage` 让 AI 决定**(代码中 `commands.ts:run`) |
| **用户中断** | run-level abort | run-level abort + 节点级 skipNode(更细) |
| **90% budget 警告** | 无公开 | **显式**:`budget-warning` 消息 + `_budgetWarningSent` 防重复 |

### 2.4 重要领域模型差异

#### 2.4.1 状态机差异

**Claude Code 推断状态机**(基于公开材料):

```typescript
// 推断(未公开)
type Status = "pending" | "running" | "paused" | "completed" | "failed" | "aborted";
// 可能还有 "budget_limited"(从"token 大幅消耗"推断)
```

**pi-workflow 显式状态机**(`state.ts:51-58`):

```typescript
type WorkflowStatus =
  | "created"
  | "running"
  | "paused"
  | "completed"
  | "failed"
  | "aborted"
  | "budget_limited"        // ← 单独
  | "time_limited";         // ← 单独

VALID_TRANSITIONS = {
  created: ["running"],
  running: ["paused", "completed", "failed", "aborted", "budget_limited", "time_limited"],
  paused: ["running", "aborted"],   // ← 唯一双向
  completed/failed/aborted/budget_limited/time_limited: []  // ← 5 个终态
};
```

**关键差异**:
- pi-workflow 把"被预算杀死"和"主动终止"严格区分(`budget_limited` / `time_limited` / `aborted` 各是终态,语义清晰)
- pi-workflow 的 `created` 是显式状态(可由 `create` action 进入),CC 推测直接 running
- pi-workflow 的状态机转移由 `transitionStatus()` **强制**(所有变更走它,非法转移 throw)

#### 2.4.2 CallCache 模型差异

| 维度 | Claude Code | pi-workflow |
|------|------------|-------------|
| **Key** | 推断 callId | `callId: number` 单调递增 |
| **Value** | 推断 AgentResult | `AgentResult { content, parsedOutput, usage, durationMs, error }` |
| **持久化** | 推断 session 内存 | **持久化**到 JSONL(随 instance 一起) |
| **跨 session 复用** | **不支持** | **支持**(resume 时按 callId 回放) |
| **节点级失效** | 推断 | `retryNode` 移除 cache,`skipNode` 注入 placeholder |

#### 2.4.3 Schema 模型差异

| 维度 | Claude Code | pi-workflow |
|------|------------|-------------|
| **格式** | JSON Schema | JSON Schema(简化) |
| **解析** | 推断用 zod | `JSON.parse` 简单 |
| **失败行为** | 推断 retry | `parsedOutput = undefined`,`output` 保留字符串 |
| **Prompt 注入** | 推断 | `agent-pool.ts:209-218` 显式在 prompt 前置指令 |

#### 2.4.4 持久化模型差异

**Claude Code(推断)**:
```
.claude/workflows/<name>.js      ← workflow 脚本(持久)
[内存] WorkflowRun 实例         ← 运行时(不持久)
```

**pi-workflow(显式)**:
```
.pi/workflows/<name>.js         ← workflow 脚本(saved)
.pi/workflows/.tmp/<name>.js    ← 临时脚本(AI 生成,未 save)
~/.pi/agent/workflows/<name>.js ← 个人脚本
[Session JSONL]
  ├── workflow-state entry       ← run 实例(每次变更追加)
  └── workflow-trace entry       ← trace 节点(每次变更追加)
```

**关键差异**:
- pi-workflow 把 run 实例的**每次状态变更**都 append 一条 JSONL entry
- rehydrate 时按 `runId` 自动 dedup(后写覆盖前写)
- **GC 是已知 trade-off**:JSONL 会膨胀,无清理机制

## 三、为什么有这些差异(设计哲学层)

### 3.1 Claude Code 的设计目标(推测)

- **超大规模**:Bun 案例是 75 万行 / 数百 agent,目标是用一台机器尽可能多地并行
- **无状态优先**:每个 workflow 是 fire-and-forget,不假设长生命周期
- **闭源 + 服务化**:与 Claude API / Bedrock / Vertex AI 集成,商业模式驱动
- **生态简单**:关键词触发 + ultracode 模式,降低用户门槛

### 3.2 pi-workflow 的设计目标(明确)

- **可恢复优先**:`ADR-002` (goal 7 态状态机) + `ADR-003` (evidence-based completion) 体现同一哲学
- **严格控制**:7 态状态机 + 3 重 budget + 节点级 retry/skip
- **开源 + 自托管**:与 Pi 平台 / xyz-agent GUI 深度集成
- **AI 辅助生成 + 显式确认**:`/workflow-generate` 三步式,不让 AI 自主写脚本
- **预算意识**:CC 默认"放飞"到 1000 agent,pi-workflow 默认"严控"到 4 并发 + 3 重 budget

### 3.3 设计目标对比

| 目标 | Claude Code | pi-workflow |
|------|------------|-------------|
| **首要 KPI** | 单次运行能解决多大的问题 | 单次运行能多严格地控制 |
| **用户假设** | 信任 AI,fire-and-forget | 信任 AI 但需要 audit trail |
| **资源假设** | 有 1000+ agent 预算 | 4 agent 并发够用,要严格计费 |
| **生命周期** | 单 session 完成 | 跨 session 持久 |
| **失败哲学** | 重跑整个 | 部分 retry,只重试失败的节点 |
| **UI 哲学** | 进度面板足够 | 进度 + 详情 overlay + GUI 描述符 |
| **AI 哲学** | 关键词触发,自动写 | 显式三步,用户确认 |

### 3.4 一个比喻

> 把 workflow 想象成"派遣团队去执行任务":
>
> - **Claude Code 像美军**:一次投送,目标明确,规模大,装备精良,但"回来需要重新部署"
> - **pi-workflow 像特种部队**:有详细 SOP,每个成员状态可查,可以中途叫停/换人/补员,任务可以跨天继续

## 四、互不可替代的部分

### 4.1 Claude Code 独占(pi-workflow 没有)

1. **官方 SaaS 集成**:Claude API / Bedrock / Vertex AI / Foundry 一键接入
2. **公开的 1000+ agent 规模验证**:Bun 案例是公开 benchmark
3. **官方超大规模 SLA**:Anthropic 兜底,服务可用性保证
4. **关键词触发 + ultracode 自动规划**:零用户操作成本
5. **审批模式生态**:Default / accept-edits / Auto / Bypass permissions 四档
6. **GitHub Actions 集成**:作为 CI 步骤

### 4.2 pi-workflow 独占(Claude Code 没有)

1. **跨会话恢复**:run 可暂停数天,resume 时按 callId 回放
2. **7 态精确状态机**:`budget_limited` / `time_limited` 单独终态,语义清晰
3. **三重预算 + 90% 警告**:token / cost / time 三轴控制
4. **节点级 retry / skip**:`retryNode(runId, callId)` 只重试一个,`skipNode(runId, callId)` 跳过
5. **AI 显式生成三步式**:`/workflow-generate` + tmp dir + `/workflow save`,用户可控
6. **GUI 渲染描述符**:`_render: { type, data }` 嵌入 details,适配 xyz-agent Vue 组件
7. **TUI 详情 overlay**:`Ctrl+O` 展开 workflow 详情
8. **节点级 pi 进程隔离**:每个 agent call 真实 `spawn("pi", ...)`,进程级隔离
9. **临时 / 已保存双目录**:`.tmp/` 临时 + saved/ 已确认,优先级 tmp > project > user
10. **白盒可调试**:每个边界(消息协议、状态机、persistence)都可观测
11. **与 Pi 平台其他扩展同进程**:与 goal / todo / coding-workflow 无缝协作
12. **持久化的 run 历史**:JSONL 留痕,审计友好
13. **Budget 警告预算**:`_budgetWarningSent` 防止 90% 警告重复发送

### 4.3 看似类似但本质不同的能力

| 能力 | 表面 | 实际 |
|------|------|------|
| **持久化** | 双方都"持久化 workflow 脚本" | CC 不持久化 run,pi 持久化 run + trace |
| **恢复** | 双方都"支持 resume" | CC **同 session**,pi **跨 session** |
| **预算** | 双方都"限制 agent 数量" | CC 16/1000 硬限制,pi 4 默认 + 3 重软预算 + 90% 警告 |
| **生成** | 双方都"AI 写 workflow" | CC 关键词自动,pi `/workflow-generate` 显式三步 |
| **Schema** | 双方都"结构化输出" | CC 推断用 zod,pi 用 JSON.parse(简单但不够健壮) |

## 五、给 pi-workflow 演进的具体建议

> 这是反向利用:既然 CC 公开了 6 月最新产品形态,我们可以吸收它的优秀设计。

### 5.1 短期可做(1-2 周)

| 建议 | 理由 | 工作量 |
|------|------|--------|
| **加 Approval Gate** | CC 第一次运行时让用户确认,降低意外消耗 | 1-2 天 |
| **加 maxAgents 硬限制** | CC 公开 1000/run 防 runaway,pi-workflow 缺这层 | 1 天 |
| **parallel 支持函数** | CC 支持 `parallel(results => ...)`,pi-workflow 只接数组,不够灵活 | 1 天 |
| **Schema 解析升级到 zod** | pi-workflow 直接 JSON.parse,schema 错误就无 parsedOutput | 2-3 天 |

### 5.2 中期可做(2-4 周)

| 建议 | 理由 | 工作量 |
|------|------|--------|
| **AgentPool 默认并发提到 8-16** | CC 是 16,pi-workflow 是 4,在中规模场景下太保守 | 1 周 |
| **JSONL GC 机制** | pi-workflow 无 GC,长 session 膨胀严重 | 1-2 周 |
| **关键词触发** | CC 用 prompt 包含 "workflow" 触发,pi-workflow 必须显式命令 | 3-5 天 |
| **加 Bundled workflow** | CC 有 `/deep-research`,pi-workflow 没有任何内置示例 | 1 周(写 2-3 个典型 workflow) |
| **优化 Meta 提取** | 每次 `loadWorkflows` 都临时 Worker,冷启动慢 | 1 周(改用静态 import 或缓存更多) |

### 5.3 长期可做(1-3 月)

| 建议 | 理由 | 工作量 |
|------|------|--------|
| **支持多 Runtime 通信** | 多个 workflow run 共享数据(目前每个 Worker 独立) | 1-2 月 |
| **可视化 Trace 工具** | CC 推测有 GUI 详情,pi-workflow 只有 TUI | 1 月 |
| **Workflow-to-Workflow 嵌套 API** | CC 文档提到"workflow 内嵌套子 workflow",pi-workflow 无显式 API | 2-3 周 |
| **Test 体系** | pi-workflow 3930 行 0 单测,风险大 | 1-2 月 |

### 5.4 不要做的

| 建议 | 理由 |
|------|------|
| **学 CC 的 1000 agent 上限** | pi-workflow 哲学是"严格控制",放飞 1000 agent 违背初衷 |
| **学 CC 的"同 session 可恢复"** | pi-workflow 已经支持跨 session,这是**优势** |
| **完全照搬 CC 关键词触发** | pi-workflow 已经有显式 `/workflow-generate`,保持显式度更符合哲学 |

## 七、总结

**Claude Code Dynamic Workflows**:
- **定位**:Anthropic 官方的 SaaS 化多 Agent 编排,主打**超大规模 + 服务化**
- **优势**:规模(1000 agent/run)、生态(SaaS 集成)、公开案例(Bun)、简单(关键词触发)
- **劣势**:跨 session 不支持、不可调试、预算粒度粗

**Pi Workflow**:
- **定位**:Pi 平台自托管的多 Agent 编排,主打**严格控制 + 可恢复 + 可观测**
- **优势**:跨 session 恢复、7 态状态机、3 重预算、节点级控制、白盒可调试
- **劣势**:默认并发低(4 vs 16)、无 SaaS 集成、无公开 benchmark、AI 生成 UX 不如 CC 简洁

**两者的最佳实践结合**:
- CC 的 SaaS 化 + 规模感
- pi-workflow 的状态机 + budget + 跨 session

如果未来 pi-workflow 要"对标"CC,核心是**提高默认并发、加 Approval Gate、加关键词触发、加内置 workflow 示例**,但**保持跨 session 恢复和精确状态机作为差异化**。
