# Workflow 下一步行动与决策摘要

> 调研链时间线: 01-调研范围与方法 → 02-生态对比 → 03-能力矩阵 → 04-功能差异对比 → 05-结论与建议 → 06-Claude-Code-Workflow-TUI → 07-下一步行动与决策(本文档)

## 背景

调研基线(05-结论与建议.md)列出 P0/P1/P2 三大类问题:
- **P0** — 立即修复:JSONL 主 session 膨胀(P0-1)、Approval Gate 失效(P0-2)
- **P1** — 中期改进:Verification 缺失(P1-1)、无 maxAgents 保护(P1-2)
- **P2** — 长期演进:nested workflow、JSONL GC 等

本文档对应 4 项 P0/P1 修复 + 1 项 doc 沉淀,**5 项决策摘要**。完整 spec 见上方 related_spec 链接。

## 5 项决策摘要

### 决策 1:External State Pointer 替代 JSONL GC

**Why:** 主 session JSONL 累积 `workflow-state` 大对象(每个 ~5-10KB),长 session 易破 100KB。传统 GC 方案需要识别"过期 entry"规则复杂、易误清。

**How:** 主 JSONL 只存轻量 pointer(`{runId, path, updatedAt}`,< 200B),完整 state 写入 `<sessionDir>/workflow-state/{runId}.jsonl`(append-only,每行一个 SerializedWorkflowInstance)。重建时从 pointer 反向读外部文件。

**Out-of-scope:** 真正的 JSONL GC 工具(超出本 spec 范围,留作 P2 后续)。

### 决策 2:True UI Approval Gate 替代 AI 自治

**Why:** 旧 `auto` mode 走 `pi.sendUserMessage` 让 AI 自治决定,易被 AI 误触发耗 budget。`force` mode 是真行为但无 UI 阻塞,用户失去拒绝机会。

**How:** `ctx.ui.confirm` 真实 TUI 弹窗,用户 y/n 决定。`sessionApprovals: Set<string>` 记忆本 session 确认过的 workflow,跨 `session_start` 通过 `workflow-approval-memory` entries 重建。tmp workflow 永远弹 confirm(不进 memory)。force mode 跳过 confirm 但 `details.confirmSkipped: true` 显式标注。

**Out-of-scope:** `auto` / `force` 重命名为 `confirm` / `skip-confirm`(向后兼容考虑,留作未来)。

### 决策 3:Verification Gate 走提示词注入(无 orchestrator hook)

**Why:** 用户明确约束 "no orchestrator hook"(避免改变 Pi agent loop 设计)。但 Verification 缺失是 P1 关键问题。

**How:** 仅更新 2 个 prompt 源:`workflow-script-format` SKILL.md 加 "Verification Patterns" 章节(Pattern A node-internal + Pattern B follow-up),`tool-generate.ts` `promptGuidelines` 数组加 1 条 verification 规则。AI 在 `workflow-generate` 时读到这些,在写 workflow 脚本时主动加 verify 节点。

**Out-of-scope:** Runtime 强制 verification 验证、verify node 失败的自动 retry 机制(超出无 hook 约束)。

### 决策 4:Soft 500 maxAgents 警告(per-workflow 累计)

**Why:** 用户用 `pi-workflow` 跑大 workflow 时,无预算耗尽前的预警。硬阈值拒绝会破坏 workflow 设计意图。

**How:** `AgentPool` 加 `totalCallCount` + `softWarningSent` 状态,`SOFT_MAX_AGENTS_WARNING = 500` 常量阈值,`onSoftLimitReached` callback 通过 `AgentPoolOptions` 注入(不直接持 `ExtensionAPI` 引用)。Orchestrator 注入 callback 调 `pi.sendUserMessage` 推 warning 到主对话流。Cache hit 不计数(只 real spawn 计数)。每个 AgentPool 实例独立 counter。

**Out-of-scope:** 硬 maxAgents 上限(留作 P2 长期,需配合 user override 配置)。

### 决策 5:文档沉淀 + CONTEXT.md 增量

**Why:** 6 个月后回看代码的人需要找到"为什么这么改"的决策链。纯 git log 不足以让新成员理解设计权衡。

**How:** 本 spec 完整存档(`.xyz-harness/2026-06-04-workflow-storage-and-verification/spec.md`) + 本文档(决策摘要 + 链接)。`CONTEXT.md` 增 4 个新术语(External State Pointer / State-Lost / Approval Memory / Verification Strategy)。

**Out-of-scope:** 单独 ADR(`docs/adr/` 暂不新增,理由见下)。

## Out-of-scope 列表

以下功能**故意**未在本次 spec 范围内,留作未来:

1. **Nested workflow**(workflow 中调 workflow) — 涉及 worker_threads 嵌套隔离,需要重新设计 orchestrator 状态机,工作量 ~2 周。
2. **硬 maxAgents 上限** — 需配合 user override 配置 + 异常恢复策略,不是简单添加 throw。
3. **`auto` / `force` 重命名为 `confirm` / `skip-confirm`** — 向后兼容考虑,留作非破坏性 API 演进。
4. **真正的 JSONL GC 工具** — 需要识别"过期 entry"规则 + 用户主动触发 + 安全网,留作 P2 长期。

## Why no ADR

按"决策是否难以逆转 / 无上下文会惊讶 / 真实权衡"三条件评估 5 项决策:
- 决策 1-4 全部**可逆**(改回旧实现只需 git revert)
- 决策 1-4 不存在"无上下文会惊讶"问题(spec 完整说明)
- 决策 1-4 不存在"真实权衡"问题(都是单选最优解)

→ 5 项决策均不满足 ADR 全部三条件 → 不创建 `docs/adr/` 新文件。如果未来某决策不可逆或引发争议,再补 ADR。
