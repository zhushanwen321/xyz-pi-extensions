# 后端编码规范

## 分层架构

基于 Clean Architecture 的 6 层划分，从外到内：

| 层 | 职责 | 依赖方向 |
|---|------|---------|
| 入口层 | 接收外部请求、参数校验、响应格式化 | 可调编排层 |
| 编排层 | 业务流程编排、事务边界管理 | 可调领域层 |
| 领域层 | 核心业务逻辑、领域模型、业务规则 | 可调数据层（通过接口） |
| 数据层 | 数据存取、ORM/查询映射 | 实现在外层，接口在领域层 |
| 集成层 | 外部服务调用、超时、降级 | 横切，可被任意层调用 |
| 基础设施层 | 配置、中间件、工具函数 | 横切，可被任意层调用 |

### 分层依赖规则

1. 外层可调用内层：入口 → 编排 → 领域 → 数据（单向依赖）
2. 内层不可调用外层：领域层不能调用入口层，数据层不能调用编排层
3. 跨层调用必须通过接口（依赖倒置）

## 编码原则

1. **最小实现**：只实现 spec 和 plan 要求的内容，不做额外优化
2. **使测试通过**：不修改已有测试文件。测试是契约，代码使测试通过
3. **TDD 顺序**：先写失败测试 → 验证失败 → 写最小实现 → 测试通过 → 重构
4. **上下文隔离**：只依赖主 AI 传入的文件路径和指令，不猜测未提供的上下文
5. **提交前自审**：
   - 所有预先存在的测试通过了？
   - 这是最小实现（无多余功能）？
   - 遵循了项目的编码规范（CLAUDE.md）？
   - 没有修改任何测试文件？

## 出现问题时的处理

- `needs_context`：缺少必要信息来完成任务（说明缺少什么）
- `blocked`：无法完成（spec 矛盾、基础设施缺失、需要超出 scope 的重构）
