3. 高峰期策略 core

3.1 时间轴预览

以一天为周期，蓝色 = 用 Z.ai（glm-5.1），橙色 = 用 opencode-go（ds-flash/ds-pro）：

Naive (当前)
0-14: Z.ai
14-18: ocg
18-24: Z.ai
peak
PeekHour
0-10: Z.ai (aggressive)
10-14: Z.ai → 混合
14-17: ocg
17-18: urgent
18-24: Z.ai
peak

↑ PeekHour-aware 在 peak 窗口也有条件地用 Z.ai（urgent / underutilized 场景），同时非高峰期更积极

3.2 决策树

┌─ computeRecommendation(config, snapshot, scene, currentModel, now) │ ├─ Layer 1: 场景硬性需求 │ scene=vision? → mimo (hard pin) │ scene=planning? → glm-5.1 / ds-pro (reasoning) │ ├─ Layer 2: KV Cache 粘性 │ 连续 ≥3 turn + ≥20k input? → stay on current (除非硬性场景不匹配) │ compaction 后 1 turn? → 自由切换 │ └─ Layer 3: PeekHour 预算决策 ← 本轮重写重点 │ ├─ 非高峰期 (0-14 / 18-24) │ ├─ zai.pct ≥ 95%? → ocg (窗口快满了) │ └─ default → zai (aggressive) │ └─ 高峰期 (14-18, 3x cost) │ ├─ zai.resetSec < 3600 && zai.pct < 85%? │ → urgent: 用 zai (窗口即将释放) │ ├─ ocg.rollingPct ≥ 80% || ocg.weeklyPct ≥ 80%? │ → fallback: ocg 快满，用 zai │ ├─ willBeUnderutilized(zai, now)? │ → 战略性: 用 zai (长期看用不完) │ └─ default → 省 zai, 用 ocg

3.3 underutilized 预测逻辑

核心问题："还剩 XX% 的 Z.ai 5h 配额，按当前消耗速率，在窗口释放完成前能不能用完？"

willBeUnderutilized(zai, plan, now): // zai.pct: 当前 5h 窗口使用率 (0-100) // zai.resetSec: 剩余多少秒首批额度释放 // rollingWindowHours = plan.rollingWindow || 5 // 如果窗口快重置了 (resetSec < 1h) 且还有余额 → 先用起来 if zai.resetSec < 3600 && zai.pct < 85: return false // 紧急消耗，不是 underutilized // 剩余时间 = 距离下一次重置 + 窗口完整周期 windowSizeSec = rollingWindowHours * 3600 remainingSec = zai.resetSec // 到第一批释放的秒数 totalRemainingBudget = 100 - zai.pct // 临界值：如果按 3x 成本消耗，在 peak 用 zai 是否划算 // 当前剩余额度 < 窗口剩余时间能自然释放的额度 → 会浪费 // 简化：zai.pct < 20% 意味着有很大空余 → 战略性使用 return zai.pct < 20

更精确的版本可以用 session entries 统计历史消耗速率（tokens/hour），但初期用简单阈值就够了。

6. 场景走查 walkthrough

场景 A：早上 10:00，coding 场景

输入: zai.pct=45%, zai.resetSec=18000 (5h) ocg.rollingPct=30%, ocg.weeklyPct=20% now=10:32, 非高峰期 决策: 第 1 层: coding → 非硬性场景，继续 第 2 层: 刚切换模型，不粘 第 3 层: 非高峰期 → zai.pct=45% < 95% → 推荐 glm-5.1 理由: "Non-peak hours, budget sufficient for Z.ai" 输出注入: Status: Off-peak hours | Current: 10:32 Z.ai: 45% [5h: 2h15m remaining, reset in 5h00m] opencode-go: rolling 30%, weekly 20%, monthly 8% >>> Recommended: zhipu/glm-5.1-plus

场景 B：下午 15:30，高峰期，ocg 即将耗尽

输入: zai.pct=60%, zai.resetSec=7200 (2h) ocg.rollingPct=85%, ocg.weeklyPct=78% now=15:30, 高峰期 决策: 第 3 层: 高峰期 → ocg.rollingPct=85% ≥ 80% → fallback to zai 理由: "Peak hours but opencode-go near limit, staying on Z.ai" 输出注入: Status: Peak hours (15:30, 3x Z.ai cost until 18:00) | Current: 15:30 Z.ai: 60% [5h: 2h00m remaining, reset in 2h00m] opencode-go: rolling 85%, weekly 78%, monthly 30% >>> Recommended: zhipu/glm-5.1-plus (Peak hours but opencode-go near limit)

场景 C：下午 17:15，高峰期，window 即将释放

输入: zai.pct=70%, zai.resetSec=2700 (45min) ocg.rollingPct=40%, ocg.weeklyPct=35% now=17:15, 高峰期 决策: 第 3 层: 高峰期 → zai.resetSec=2700 < 3600 && zai.pct=70% < 85% → urgent: 用 zai (窗口 45min 后释放, 还有 15% 余额) 理由: "Urgency: Z.ai window resetting soon with sufficient budget" 输出注入: Status: Peak hours (17:15, 3x Z.ai cost until 18:00) | Current: 17:15 Z.ai: 70% [5h: 1h30m remaining, reset in 45m] opencode-go: rolling 40%, weekly 35%, monthly 10% >>> Recommended: zhipu/glm-5.1-plus (Urgency: reset soon)

场景 D：下午 16:00，高峰期，所有条件正常

输入: zai.pct=30%, zai.resetSec=14400 (4h) ocg.rollingPct=45%, ocg.weeklyPct=30% now=16:00, 高峰期 决策: 第 3 层: 高峰期 → zai.resetSec=14400 > 3600, 不 urgent → ocg.rollingPct=45% < 80%, ocg 充裕 → willBeUnderutilized: zai.pct=30% < 20%? → No (30% > 20%) → 省 zai, 用 ds-flash 理由: "Peak hours, saving Z.ai quota for later" 输出注入: Status: Peak hours (16:00, 3x Z.ai cost until 18:00) | Current: 16:00 Z.ai: 30% [5h: 3h30m remaining, reset in 4h00m] opencode-go: rolling 45%, weekly 30%, monthly 12% >>> Recommended: router-openai/ds-flash (Peak hours, saving Z.ai quota)

场景 E：凌晨 02:00，深夜 coding，zai 快满

输入: zai.pct=97%, zai.resetSec=600 (10min) ocg.rollingPct=50%, ocg.weeklyPct=40% now=02:00, 非高峰期 决策: 第 3 层: 非高峰期 → zai.pct=97% ≥ 95% → zai 窗口几乎满了, 切 ocg 理由: "Non-peak, Z.ai rolling window nearly full, switching to opencode-go" 输出注入: Status: Off-peak hours | Current: 02:00 Z.ai: 97% [5h: 0h09m remaining, reset in 10m] opencode-go: rolling 50%, weekly 40%, monthly 20% >>> Recommended: router-openai/ds-flash (Z.ai window nearly full)
